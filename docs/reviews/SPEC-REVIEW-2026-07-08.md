# Adversarial Spec Review — claude-fleet spec ecosystem

Date of review: 2026-07-08
Scope: docs/SPEC.md (v2, post F1–F16 appendix), docs/ROADMAP.md, all seven docs/specs/*.md stubs, docs/PRIOR-ART.md, docs/IDEA-FORGE-REPORT.md §5 graveyard — cross-checked against the shipped implementation (bin/fleet.py @ 792583c on branch fleet-impl, bin/hooks/*, tests/ [338 unit tests], the Campaign-0 live smoke report at .superpowers/sdd/integration-smoke.md, knowledge/lessons.md).
Method: multi-reviewer adversarial pass with independent verification of every finding. Findings that restated the settled SPEC appendix items F1–F16 or re-proposed IDEA-FORGE §5 graveyard ideas were killed in verification. Final tallies: **30 confirmed (5 blockers, 25 majors), 25 unverified (verification budget exhausted — retained at lower confidence), 6 refuted** (disposition appendix, with reasons, so the next review does not re-litigate them).

## 1. Executive summary

The spec ecosystem is **not yet fit to hand to the fleet for self-build**. The core problem is not design quality — the v2 design survived its adversarial review and the shipped Phase-1 code is largely correct — it is that **SPEC.md has fallen behind its own implementation**, and every later phase is instructed (ROADMAP L101) to inherit the spec text verbatim. The correct semantics (sticky dead/attached, launch pre-claim, hook-tail classification, cost baseline, respawn/kill guard ladders, registry quarantine) live only in bin/fleet.py docstrings and knowledge/lessons.md; the normative document still describes the pre-Campaign-0 behavior that the live smoke proved wrong. A Phase-2 watchtower built against SPEC §4 as written would resurrect operator-killed workers, treat every healthy completed turn as dead, and launch second live claude processes onto sessions mid-launch.

Five blockers must be resolved before any phase stub is claimed for drafting:

1. **SPEC §4/§5/§11 status state machine amendment** (B1): sticky dead, sticky attached, pre-claim scoping, last-substantive-event classification, interrupt/kill turn_pid semantics, reboot row.
2. **Log-rotation vs. long-lived reader handle discipline on Windows** (B2): a fixed constraint for the watchtower and web-UI stubs, plus a retry/rollback clause in SPEC §5/§11 — otherwise a log tailer vetoes respawn, inverting the "core never depends on any surface" principle.
3. **Portability scope must include porting the test suite** (B3): the phase's own done criterion ("tests green on all 3 OSes") is unachievable against the Windows-pinned tests as scoped.
4. **Manager-on-call must be a registry entry** (B4): otherwise the one-live-claude and cwd-scoped-resume invariants are unenforced on the highest-privilege path, and concurrent triggers (two Telegram messages; watchtower wake-up) race with no claim protocol.
5. **Manager-on-call privilege bounds promoted to fixed constraints** (B5): never bypass; transitive-privilege confirmation round-trip for any path that mints a bypass worker; /kill and /respawn confirmed.

What must be amended first, in order: (a) a SPEC v2.1/v3 amendment pass folding the spec-vs-code drift items (§4 of this report) into normative text, appendix-style, matching the F1–F16 disposition format; (b) fixed-constraint injections into the watchtower, telegram, portability, and providers stubs for the blockers and the watchtower/providers majors; (c) two scope decisions (cut-or-cap auto-respawn; token-ceiling enforcement lever) that gate Phases 2 and 5; (d) a testing-infrastructure decision (scripted live-integration tier + fixture corpus) that three later phases silently depend on. Until (a) lands, "inherit SPEC.md verbatim" propagates known-wrong ground truth.

## 2. Confirmed findings

### 2.1 Blockers

**B1. SPEC §4 status state machine is stale and unscoped versus shipped code** — *SPEC §4 recompute rules, §5 interrupt/kill rows, §11 reboot row; inherited by phase-2-watchtower OQ1 and ROADMAP Phase 2 --auto-respawn.*

One root defect with five faces. (a) Sticky dead: §4/§11 as written resurrect an operator-killed worker (kill → dead; log tail is a result event → next recompute flips to idle), making kill non-terminal and `fleet clean` unreachable; shipped code makes dead sticky with respawn the only exit (SMOKE-D, commit 641298d; pinned by tests/test_core.py:497). (b) Sticky attached: literal §4 demotes attached to idle (stale turn_pid + result tail), after which a send launches a resume turn against a session that may still hold a live TUI — violating one-live-claude; §11's reboot row ("statuses recomputed from log tails," L194) contradicts §5/§9's explicit-release model. (c) Hook-tail: §6 mandates `--include-hook-events`, and hook/system lines land AFTER the result line on every real launch (SMOKE-A), so §4's literal "last stdout line" rule (L80) classifies every healthy completed turn as dead; shipped `_last_line_type` scans backwards past `type:"system"` lines. (d) Pre-claim: status=working with turn_pid=None is written under fleet.lock before Popen by every turn-starting path and must never be demoted — this is what enforces one-live-claude against concurrent send/attach; §4 gives turn_pid no null-while-working meaning, and the killed-CLI pinned-working residual is undocumented. (e) Interrupt contradiction: §5's interrupt row says "→ idle" but §4's recompute of the killed turn's stale PID with no result line yields "dead"; nothing says which write wins or that interrupt/kill clear turn_pid.

*Failure scenario:* any spec-conformance test fails against correct code; a Phase-2 auto-respawn built against §4 as written resurrects killed workers, treats the pre-claim window as a crashed turn, marks every finished worker dead (notification storms), and respawns healthy workers — launching a second live claude onto a mid-launch session. *Amendment (SPEC.md §4/§5/§11/§12 + phase-2-watchtower.md):* issue a SPEC v2.1 amendment, appendix-style: (1) recompute applies only to records in status working with non-None turn_pid; (2) dead is sticky (respawn the sole exit), attached is sticky (release or respawn --force the sole exit); (3) working+turn_pid=None = launch-in-flight pre-claim, never demoted; document the killed-CLI residual; (4) classification = last substantive stream event, scanning past `type:"system"` and junk; (5) interrupt/kill clear turn_pid on commit; (6) §11 reboot row exempts attached (recovery = manual release + doctor nag). Add these as named required regressions in §12. Add a fixed constraint to phase-2-watchtower.md: rules MUST be written against §4 as amended, never raw PID/tail heuristics, and auto-respawn must distinguish crash-dead from operator-kill-dead before keying on dead status.

**B2. Log rotation via os.replace is incompatible with watchtower/web-UI log tailing on Windows** — *SPEC §5/§11 rotation vs phase-2-watchtower watch loop and phase-4-webui peek SSE (OQ3).*

`cmd_respawn` rotates logs with a bare `os.replace` (bin/fleet.py _rotate_worker_log, no exception handling, called AFTER the new-sid registry record is committed). On Windows, os.replace raises PermissionError while any process holds the file open (CPython `open()` omits FILE_SHARE_DELETE — empirically reproduced during verification with py -3.13 on this machine). The watchtower is specified as "a READER of logs/registry" that "tails logs/*.jsonl" (stub L20, ROADMAP L34), and phase-4 OQ3 proposes an SSE follow mode reusing peek's parser — a hold-open handle by design; neither stub mentions file-handle discipline (watchtower OQ3 discusses only polling cost). A naive tailer makes every respawn fail while it runs — including the watchtower's own flagship auto-respawn, and a respawn triggered from the web UI's own button while its peek pane is open. That is a surface vetoing a core operation, inverting ROADMAP principles #1/#2; Phase 2's done criterion covers "CLI unaffected with watchtower stopped" but not the real hazard, CLI broken with watchtower RUNNING.

*Amendment:* fixed constraint in phase-2-watchtower.md and phase-4-webui.md — all log/state readers open-read-seek-close within each poll cycle; no handle outlives a poll tick; tailers tolerate replacement between polls (size shrink / file-ID change → reset cursor). Amend SPEC §5 respawn / §11 log-bloat rows: rotation retries briefly on PermissionError and fails the respawn cleanly with rollback via `_unrotate`, naming the likely holder. Add a test pinning "respawn succeeds while a follower is mid-poll."

**B3. Portability done criterion "tests green on all 3 OSes in CI" is unachievable as scoped** — *docs/specs/portability.md Scope (L12) + Done criteria (L34).*

The stub's scope (adapter, init, FLEET_HOME, shims, CI matrix, docs) omits test-suite work, but tests/test_hooks.py:27 hardcodes `PY_CMD = ["py", "-3.13"]` (all hook tests error on POSIX); test_steering.py:768 and test_cli.py:273 assert Win32-only subprocess constants (`DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP` — AttributeError on POSIX); and test_steering.py:828 `test_posix_platform_raises_unsupported` pins the exact behavior the portability build must invert. OQ8 asks only which tests can run without a claude binary. Two implementers resolve differently: one skip-marks everything on POSIX (green CI, zero POSIX coverage, criterion vacuously satisfied), the other parametrizes per OS.

*Amendment:* add to portability.md scope: port the test suite — hook tests invoke via sys.executable; platform-launch assertions go through the adapter interface with per-OS expected-kwargs parametrization; invert/replace the POSIX-unsupported test with POSIX behavior tests (start_new_session, killpg); the TestPlatformAdapterBoundary lint suite must stay green unmodified. Add a resolved OQ on pytest tier markers: unit+hooks in the CI matrix; live-integration local-only, env-gated.

**B4. Manager-on-call is a resumable session outside the registry — one-live-claude and cwd-scoped resume are unenforced on the highest-privilege path, and concurrent triggers race** — *phase-3-telegram.md Scope + OQ3, ROADMAP Phase 3, phase-2-watchtower OQ8.*

One-live-claude-per-session (SPEC L117, §9 L172) is enforced only via the worker registry's pre-claim protocol; a manager session that is not a registry entry (the §4 schema tracks workers only) has no claim protocol at all. Two Telegram messages seconds apart (trivially common) launch two concurrent `claude -p --resume <manager-sid>` turns on one session — concurrent transcript writes, exactly the corruption the invariant prevents. "Cooldown-limited" (L19) rate-limits starts but decides nothing about a second message's fate while a multi-minute manager turn runs; watchtower OQ8's pre-Telegram wake-up adds a second independent trigger for the same session, so the race exists across phases. cwd-scoped resume (SPEC L21) is likewise unbound — OQ3 asks "which cwd" but nothing makes it a recorded, immutable field.

*Amendment:* fixed constraint in phase-3-telegram.md — the manager-on-call session MUST be a registry entry (reserved name, e.g. `manager`; cwd = FLEET_HOME, immutable), so every existing guard applies for free: pre-claim under fleet_lock, send-while-working → mailbox queueing (a second message becomes mail drained into the running turn or the next launch), attach guards, journal+respawn continuity. Exactly one manager turn in flight; queued messages coalesce into the next wake; cooldown = minimum gap between turn STARTS. Watchtower OQ8's wake-up (if it survives) must go through the same entry. Add an "Invariants touched" section naming one-live-claude and cwd-scoped resume.

**B5. Manager-on-call privilege bounds left open — "bypass?" is a decide-later and no confirmation round-trip is required for destructive remote commands** — *phase-3-telegram.md Fixed constraints (L19) + OQ1.*

The stub's own words call this "the highest-privilege path in the system," yet a drafting session may legitimately resolve its permission mode to bypass, and whether /kill or spawn-with-bypass needs confirmation is merely OQ1. Worse, the manager's own permission mode does not bound what it creates: a manager turn in "accept" can still run `fleet spawn x --mode bypass` via Bash, minting a `--dangerously-skip-permissions` worker. Net: compromise of the owner's Telegram account (session hijack, SIM swap, stolen phone) = silent arbitrary code execution on the dev machine — and the purely functional done criterion (phone-only campaign) exercises no security behavior, so nothing in the build process catches a catastrophic resolution.

*Amendment:* promote to fixed constraints before drafting: (1) manager-on-call turns NEVER run --dangerously-skip-permissions; (2) transitive-privilege rule — any inbound path that would create a bypass-mode worker requires an explicit owner confirmation round-trip (bot replies with exact command + nonce, owner replies confirm); (3) /kill and /respawn require the same confirm; (4) the drafting session must enumerate the exact command set allowed without confirmation. (Verification note: a one-shot nonce confirm is not graveyard #1's dead ack/suppress escalation ladder — no state machine, no escalation.)

### 2.2 Major findings

**Core SPEC (Phase 1 text).**

**M1. Cost model undocumented; no schema-evolution rule for fleet.json.** SPEC §4's schema lists only `cost_usd` (L71) with no semantics, but SMOKE-B live-proved that `--resume` turns report cost per-invocation (a later turn can report a SMALLER total than an earlier one), and the implementation computes cost_usd = `cost_baseline` (stamped by respawn, 0.0 on spawn, absent from the schema) + sum of every result event's cost in the current log (fleet.py new_worker_record, _sum_result_costs, recompute_worker; pinned by test_resilience.py TestCostBaseline). A spec-only implementer writes last-result-wins accounting — the exact live bug already fixed. Three later specs (providers OQ3 untrusted-cost marking, watchtower budget rules, web-UI spend telemetry) build on these undocumented semantics, and no additive-schema rule exists for fleet.json (cost_baseline already exists undocumented; providers adds a profile field; watchtower likely adds mute/cursor state). *Amend SPEC §4:* add cost_baseline plus a cost-model paragraph (per-invocation result costs; sum-over-current-log + baseline; baseline carried by respawn; cost_usd = lifetime spend of the NAMED worker); add an explicit additive-schema rule (readers default missing fields, writers preserve unknown fields); cross-reference from the §5 status row as the input contract for providers/watchtower/web-UI spend features.

**M2. SPEC §5 command table missing the shipped guard semantics for respawn/kill/clean.** The respawn row (L97) has no `--force` and no refusal cases, yet the implementation refuses while a turn runs unless --force (verified kill via _interrupt_worker; kill_failed aborts with no mutation), refuses attached workers without --force, and refuses launch-in-flight claims even WITH --force. Kill refuses launch-in-flight, re-probes a "not_running" verdict after 0.5 s (_DEAD_CONFIRM_DELAY_SECONDS), and marks dead but exits 1 loudly on kill_failed. Clean also sweeps rotated `.jsonl.1`/`.err.1` logs and orphaned `mailbox/*.claimed.*` files. Walking respawn-against-a-live-turn under the spec as written leaves everything undetermined: rotation renames a log the live claude holds open, the old turn's hooks race respawn for the old sid's mailbox, and two live fleet-launched claudes end up under one worker NAME (the §6 guard is phrased per-session, so this skates by on a technicality while defeating its purpose). A Phase-2 auto-respawn rebuilt from §5 rebuilds these races. *Amend:* one sentence each in the §5 respawn, kill, and clean rows recording the --force ladder (rotation only after the old turn is confirmed dead), in-flight refusals, dead-confirm double-probe, kill_failed exit contract, and the full clean artifact sweep — the table is the contract build sessions read.

**M3. Respawn's sid-swap+drain is not serialized against a concurrent `fleet send` — mail orphaned forever on a dead sid.** The §4 lock guards only fleet.json writes; a send to a working/attached worker resolves the sid under fleet_lock but appends to the mailbox OUTSIDE it (cmd_send), with no post-append registry re-check. Interleaving: respawn swaps the record to new sid S2 under the lock and drains/removes mailbox/S1.md; the in-flight send then appends, recreating mailbox/S1.md — which no drain rule ever touches again (the universal drain rule and the idle+mail flag are both keyed to the CURRENT sid). The steering message is silently lost, contradicting §5's "consumed, not orphaned" promise (L97). Phase-2 auto-respawn initiating respawns asynchronously makes this manager-vs-machine window common. Doctor's "orphaned/pending mailboxes" check (L99) detects but has no defined term or recovery lever. Distinct from settled F9 (drain-at-respawn-time — this is mail arriving after that drain); not graveyard #8 (no DLQ — this is a lock-ordering rule). *Amend SPEC §5/§4:* send's working/attached-branch mailbox append happens under fleet.lock (or send re-reads the registry after append and re-appends to the new sid on change); state respawn's drain+sid-swap is atomic w.r.t. send; phase-2-watchtower fixed constraint: auto-respawn goes through cmd_respawn under fleet_lock; define doctor's "orphaned mailbox" (sid matches no registered worker) with a stated disposition (print sid + first line; operator re-sends or deletes).

**M4. PID liveness probe cannot distinguish "gone" from "exists but unreadable," and no failure direction is specified.** The probe is one PowerShell call interpolating `$p.StartTime` with no try/catch; for a higher-integrity process (worker spawned from an elevated terminal, `fleet status` run unelevated) Get-Process returns the object but .StartTime throws Win32Exception — stdout empty, get_process_info → None, pid_alive → False. Any transient query failure, or a null/corrupt stored turn_pid_ctime (the launch path itself stores None on query failure), silently classifies a live turn as stale → mid-stream log, no result → dead (sticky) → respawn's turn-running guard sees the same false negative and Popens a second live claude while the first turn still runs; taskkill is access-denied in the same elevation scenario so interrupt cannot clean up. Kill/clean got a 0.5 s dead-confirm re-probe; ordinary status classification did not. portability.md OQ2 covers only per-OS precision, not query-failure semantics. *Amend §4:* the probe distinguishes alive-and-matching / definitely-gone / exists-but-unreadable (= alive-unknown: never demoted to dead, never respawn/kill-eligible without --force) via a try/catch marker in the PS script (optional Get-CimInstance fallback); missing pid/ctime or query failure ⇒ not-live, but retry once after a short delay before any working→dead transition; document a same-elevation operating rule + doctor check; state the accepted residual. Carry per-OS failure modes into portability OQ2.

**M5. --max-budget-usd and --setting-sources apply only to the first spawn turn.** Both flags are per-print-invocation ("only works with --print" per `claude --help`); only cmd_spawn passes them — the send-when-idle resume launch and cmd_respawn pass only model, and the §4 schema has no fields to carry them. A worker spawned with --max-budget-usd 5 and steered with three sends runs turns 2–4 uncapped while the operator believes §11's documented runaway mitigation (L196) is armed — the cap evaporates exactly on long babysat jobs, the case it exists for. Identically, §6's foreign-hostile-Stop-hook remedy (--setting-sources, L126) evaporates on turn 2 — and the foreign hook fires on EVERY turn. SMOKE-B additionally means even re-passing gives N×budget across N turns (the CLI cap is per-turn). *Amend:* add max_budget_usd and setting_sources to the §4 schema (recorded at spawn, immutable like mode); every launch path re-passes them; document in §11 that the CLI cap is PER TURN, with cumulative worker-level enforcement fleet-side (compare registry cost_usd against the cap before launching any resume turn; refuse + flag in status) or an explicit first-turn-only caveat — do not leave it implicit. Afternoon-sized; add to phase1-hardening-kernels.md.

**M6. Hook data-plane gap: ready-for-build hardening kernels violate SPEC §4's hook write boundary.** SPEC L78: "Hooks never write registry or events; they touch only mailbox files." Kernel 1 appends hook exceptions to state/hook-errors.log; kernel 3 has a PostCompact hook append to the worker's journal — both hook writes outside the mailbox, in a spec marked ready-for-build TODAY: a builder honoring SPEC §4 must refuse the kernels, one honoring the kernel spec silently breaks the boundary. Compounding: §7 and the settings template wire only PostToolUse and Stop (kernel 3 never says to register PostCompact, and kernel 2's static lint of known event names would flag the new registration); journals are keyed by NAME while hooks receive only session_id — kernel 3 needs a sid→name registry read SPEC nowhere sanctions, with no failure-tolerance rule; hook-errors.log is a new multi-writer file with no stated discipline. *Amend:* rewrite §4's invariant to the real rule (hooks never write registry or events; they may append to mailbox files, state/hook-errors.log [append-mode single-line best-effort, no lock], and their own worker's journal via PostCompact; registry access read-only and failure-tolerant, exit 0 on any failure); amend kernel 3 to add the template/§7 registration, specify sid→name resolution with a sid-keyed fallback or skip, verify the actual event name against the installed CLI (verification confirmed the installed claude 2.1.203 does ship a PostCompact hook event — record this), and teach kernel 2's lint the new name.

**M7. SPEC §12 integration testing is manual-only with no repeatable harness or fixture corpus.** The one live smoke found four bugs 338 green unit tests missed (lessons.md's sharpest Campaign-0 lesson: "340 green unit tests coexisted with three High/Medium bugs only a real haiku worker exposed"); tests/ contains zero integration files and no marker config; no phase schedules a scripted re-run — the next change to launch_turn, hooks, or stream parsers gets regression pins for the OLD findings and nothing for new ones. Every later phase leans on the missing infrastructure: watchtower OQ1 demands verifying rules "against real log samples" with no sample source anywhere in the repo, providers needs live turn verification, Phase 5 modifies the Stop hook every worker runs. *Amend §12:* add tier-3 tests/integration/ — scripted pytest gated on FLEET_LIVE=1, haiku model, temp FLEET_HOME + throwaway git repo, asserted budget ceiling (~$0.25), replaying the documented smoke scenarios (spawn/wait/result, mid-turn send injection, Stop-block dual outcome, attach guard, respawn drain+journal+cost-baseline); required before merging changes to launch_turn/hooks/stream parsers and at each phase gate (one sentence in ROADMAP discipline); each run archives sanitized stream-json logs into tests/fixtures/streams/ — the declared corpus for watchtower OQ1. Add as item 6 to phase1-hardening-kernels.md.

**M8. §12's designated first M2 test (headless Stop-block) has no pass criterion consistent with the documented drain race.** §5's universal drain rule documents that a send landing after the Stop hook drains loses the race and is delivered next launch, so "behaves as documented" has two legitimate outcomes and the spec never says which counts as PASS. The only recorded run hit exactly this: DONE_THIRD never delivered same-turn, graded "FAIL on the literal ask," then ad-hoc reclassified as the by-design race — while the race-losing path's safety net (idle+mail flag) was itself broken at the time (Findings A/C), which a dual-outcome criterion would have surfaced as the actual failure. Two future testers grade the same transcript PASS and FAIL. *Amend:* dual-outcome criterion — PASS iff EITHER (a) the log shows a Stop hook_response with `{"decision":"block"}` and the queued text in the same turn's transcript, OR (b) the message remains in mailbox/<sid>.md AND status flags idle+mail AND the next launch's composed prompt contains it. Anything else (lost, flag absent, delivered twice) = FAIL. Add a deterministic variant seeding the mailbox early in a turn guaranteed still running.

**Phase 2 — Watchtower.**

**M9. Watchtower is constrained to be "a READER of logs/registry" while its scope mandates auto-respawn — a registry-writing, turn-launching, process-killing action with unspecified invocation path and lock discipline.** The fixed constraint (stub L20) is false as written: --auto-respawn mutates the registry (new sid, status, cost_baseline), rotates logs, drains mailboxes, and Popens turns; ROADMAP L36's journal-freshness "pings worker to update first" is a send — which on an idle worker LAUNCHES a resume turn (universal drain rule); and the context-burn trigger fires mid-turn by definition, so auto-respawn must first interrupt a live turn — the shipped `respawn --force` verified-kill path §5 does not even record. The stub never says how the resident loop invokes any of this (in-process call vs shelling out), and a builder taking "READER" literally bypasses fleet_lock and the pre-claim protocol — an unlocked registry write (single-writer violation) or two turn launches on one session (one-live-claude violation). OQ4 covers journal freshness and loop caps, not invocation mechanism, lock discipline, or the mandatory mid-turn interrupt; the stub names neither invariant. *Amend:* rewrite the constraint — watchtower is a reader for RULE EVALUATION; every ACTION (send, respawn) executes only via the same lock-guarded fleet.py command paths as the CLI (direct function call within a short-lived lock scope, never holding the lock across the poll loop), inheriting pre-claim, sticky-dead, --force guards, and drain semantics. Define behavior when the guard/kill fails (emit event, back off, count toward the respawn-loop cap). Add an "Invariants touched" section naming single-writer-registry and one-live-claude.

**M10. Auto-respawn is a safety-critical autonomous-mutation subsystem in Watchtower v1 with no hard resource caps.** Phase 2's done criterion is detection ("phone-free day — nothing silently died"), but the scope mandates auto-respawn whose loop safety OQ4 holds as an unanswered question ("Cap per day?" — a question mark, not a fixed constraint) while the done criteria pre-commit to "auto-respawn demonstrated lossless on a real task" — so the stub structurally forecloses cutting it. A worker whose journal passes freshness but whose task deterministically burns context respawn-loops unattended (a weekend of 15-minute full-context cycles at sonnet prices is hundreds of dollars). Phase 3 delivers remote manual respawn — the human-vetted lever ("respawn is the only recovery lever," lessons.md) — one phase later. And no fleet-WIDE cost rule exists anywhere in the ecosystem: every budget mechanism is per-worker, while this system's failure mode is many workers each individually under threshold. *Amend (preferred):* cut auto-respawn from Phase-2 scope (v1 detects and notifies context-threshold/dead/idle-with-mail); move it to a post-Phase-3 re-vet gated on felt friction; delete "auto-respawn demonstrated lossless" from done criteria. If kept: promote a hard respawn cap per worker per rolling 24 h (exceeded → dead + notify, never silent retry) to Fixed constraints, and add a fleet-wide cumulative daily cost threshold to rule set v1 (a notification rule summing registry cost_usd — not graveyard #4's dead spend-ledger).

**M11. "Notifier plugin interface" for three known first-party sinks is framework-building.** Total notifier population across all roadmap phases: desktop (via platform adapter), file append, Telegram (Phase 3) — all first-party, all one repo, all receiving the same typed event. A plugin interface (registration, discovery, routing config, per-plugin error isolation) is a subsystem whose only consumer is the repo itself — the YAGNI trap ROADMAP principle 4 ("no framework-building") and "flag, not subsystem" prohibit; the word "plugin" in the scope line will cause an implementer to build an abstraction layer nobody outside the repo can load. *Amend:* scope says "notifier functions (desktop + file)"; resolve OQ5 as a single notify(event) iterating a static in-code sink list — no registration, no routing config file; Telegram = append one sink function; revisit only if a fourth, out-of-repo sink materializes.

**M12. Per-OS service install in Watchtower v1 is a 3-OS installer subsystem replaceable by doc snippets.** Automated install/uninstall across Windows Scheduled Tasks, systemd user units, and launchd agents — plus lifecycle commands, doctor checks, and OQ7's auto-restart story — multiplies exactly the OS surface the roadmap ships Phase 1.5 to contain ("watchtower multiplies OS surface," L22), for a single-user tool whose daemon is optional by principle 2. 80% of the value is `fleet watch` in a terminal plus three copy-paste unit/task snippets in docs. *Amend:* v1 = foreground `fleet watch` + docs/ snippets for hand-installed per-OS autostart (which preserves OS-managed crash-restart, answering graveyard #10's cause of death); doctor gains only a "watchtower heartbeat stale" NOTE; defer install/uninstall commands to a post-use-week re-vet.

**M13. A Windows run-without-logon task or true service puts the watchtower where desktop toasts are invisible — silently defeating the phase's anti-silent-failure goal.** Two Windows facts the stub records nowhere: (1) a Scheduled Task configured "Run whether user is logged on or not" (or an sc.exe/NSSM service — which OQ7's "service auto-restart" nudges implementers toward) runs in a non-interactive session where WinRT toasts never render, with no error: the watchtower detects dead workers, "notifies," and the operator sees nothing — while the phone-free-day criterion appears to pass because nothing visibly fired. (2) Python stdlib has no notification API on any OS; the no-deps Windows route is a PowerShell WinRT ToastNotificationManager shell-out that itself only works from an interactive session. Two implementers resolve differently today: one ships a logon task (works), one ships a service (silently broken). *Amend fixed constraints:* Windows target is a LOGON-TRIGGERED INTERACTIVE scheduled task (schtasks /sc onlogon, run-only-when-logged-on, current user — no /ru SYSTEM, no service wrapper); macOS = launchd user agent in the gui domain; Linux = systemd --user unit. Corollary in OQ7: watchtower is down from boot until first logon — doctor/CLI must surface "watchtower not running." Specify the Windows desktop notifier as the PowerShell WinRT toast shell-out through the platform adapter; file notifier is the always-works fallback.

**M14. `fleet conflicts` has no defined input universe.** The kernel is "a one-shot advisory git merge-tree sweep across worker branches/worktrees," but the registry's only location datum is the immutable cwd — no branch, no worktree/common-repo identity, no base branch — and the OQ list ("answer all") has zero conflicts questions, so a spec author can mark the stub complete without deciding repo grouping (cwd equality misses worktrees of one repo), ref selection (HEAD at scan time? branch at spawn?), comparison set (pairwise vs each-vs-main), or detached-HEAD/dirty-tree handling (merge-tree sees commits, not the working tree — a worker's real conflict surface is usually uncommitted). Two implementers produce incomparable tools. *Amend:* add a watchtower OQ covering all four (resolve git common dir per worker cwd; HEAD of each worker's cwd at scan time with uncommitted-blindness noted in output; pairwise among co-repo workers + each vs configured base; skip non-repo/detached with note); decide whether spawn records branch-at-spawn — if so, flag the SPEC §4 schema amendment now.

**M15. Watchtower context-% estimation is wrong twice over.** (1) Method error: each API call re-sends the whole conversation, so the usage block on the LAST result event (input_tokens + cache_creation + cache_read) directly measures current context occupancy; OQ2's cumulative input+output summation double-counts every earlier token on every subsequent turn (roughly quadratic in turn count), firing "context > X%" on a near-empty context after a handful of turns, and can never account for compaction — whereas the last-turn measure self-corrects after it, inverting OQ2's stated worry. (2) Blind spot: the attached TUI's stdout never reaches the tailed log and hooks don't fire during attach (SPEC L172), so two interactive hours can move a session 30%→90% while the estimator reads 30% — release → the next headless turn resumes a nearly-full context and the rule that exists to prevent this never fired; the done criterion is passable on the headless-only demo path. *Amend:* rewrite OQ2 as resolved-in-principle (context % = most-recent-result-event input-side usage / model window; remaining sub-question = model-name → window-size mapping without a rot-prone table); on attach mark the worker's context estimate UNTRUSTED, re-baseline from the first post-release headless turn, and add a rule state "context estimate stale (attached N h)" so the watchtower reports blindness instead of a false-low number.

**M16. The watchtower rule engine has no test strategy and its soak done-criteria are unfalsifiable.** OQ1 correctly demands verifying each rule "against real log samples before speccing detection logic," but no corpus exists and nothing is spec'd to produce one — inviting the synthetic-log false-green trap lessons.md warns about (hook events land after result lines only in REAL logs). "Zero silently-dead workers" over a week is circular (verifying the zero requires the silent-death detection under test); "before a worker degraded" has no defined "degraded." Built this way, the rule engine ships with its detection logic never exercised against a single real failure, and the phase gate is a vibe. *Amend constraints:* rules are pure functions (parsed stream events + registry snapshot → typed event or None), unit-tested against the git-tracked fixture corpus from M7; the file notifier is the test seam for fault-injection acceptance tests (kill a live turn PID → crash event within ≤1 poll interval; synthetic result events → thresholds fire at spec'd numbers; watchtower stopped → full CLI suite still green — principle #2, directly testable); the week-of-use soak stays IN ADDITION to, never instead of, the injectable tests.

**Phase 2.5 — Providers.**

**M17. providers.md's fixed mechanism "attach injects the same env into the launched terminal" is infeasible by process-env inheritance on the primary paths, the obvious workaround leaks secrets onto command lines, and secrets storage is an unresolved "or" inside a fixed section.** The natural implementation (Popen(env=merged) on the terminal launcher) silently fails on the primary Windows path: wt.exe hands the tab request to an already-running WindowsTerminal.exe monarch, so the new tab's shell inherits the SERVER's environment, not fleet.py's — and attach never checks the child by design, so there is no error (the failure only occurs when a Terminal window already exists, so even the done criterion's attach-env verification can pass in acceptance and fail in production). tmux new-window runs in the tmux SERVER's env; macOS osascript "do script" shells are parented by Terminal.app/launchd — same silent failure on the POSIX paths portability.md enumerates. The operator attaches to a proxy-profile worker and the interactive session silently talks to the default Anthropic backend on their own account — exactly the bug the stub calls "a correctness bug, not a nicety." The fix-by-string workaround (`$env:ANTHROPIC_API_KEY='sk-...'; claude --resume ...`) puts secrets in Win32_Process.CommandLine, script-block logging, and `ps auxww` — violating "no secret ever landing in git or logs." Within the same fixed section, secrets storage is left as an unresolved alternative ("state/secrets.toml (or env-var names resolved at launch)"), and providers.toml's on-disk location is never stated. *Amend Mechanism:* env delivery is a platform-adapter contract — a per-attach generated env script (0600-equivalent, under gitignored state/, deleted on release) that the launched shell sources before exec'ing `claude --resume`; secret VALUES never appear in argv, command strings, logs, events, registry, or echo output (names may). Pick one secrets storage model, specify file permissions + doctor's secret-presence check; state providers.toml's path. Add a portability OQ3 row verifying env delivery per terminal path, plus a doctor/test assertion that the attach command line contains no secrets value. Done criterion: attach-env verification specifically on the wt path with a pre-existing Terminal window open.

**M18. Behind proxy provider profiles, --max-budget-usd enforces against wrong-or-absent dollars — no functional spend bound from Phase 2.5 until Phase 5.** fleet merely passes the flag to the claude CLI, which enforces from its own Anthropic-priced accounting — which providers.md OQ3 itself concedes is "wrong or absent" behind a proxy: absent/zero → the cap never trips (worker unbounded, precisely the §11 risk, with no spawn warning, status flag, or doctor check); inflated → early mid-task trip with no defined status/result classification (OQ8 lists proxy-down/token-expiry/rate-limit, not budget-trip). OQ3's remedy covers accounting/display only; no OQ names the flag. The replacement guard — the token-ceiling kernel OQ3 itself cites — is scheduled in Phase 5, after Phases 3–4, with no dependency on either; proxy-profile workers exist precisely to run long unattended campaigns. Moving the kernel earlier is the salvage table's own sanctioned path ("or earlier as a one-liner") and does not resurrect graveyard #3 (no PreToolUse, no price table). *Amend:* move the token-ceiling kernel into providers.md scope (or phase1-hardening-kernels.md), noting in phase-5 it shipped early; providers done criteria gains "a proxy-profile worker demonstrably halted by the token ceiling"; add providers OQ9 (--max-budget-usd × untrusted proxy costs — warn/refuse the flag on non-default profiles or translate to a token ceiling; define budget-trip classification, extending OQ8); caveat SPEC §11 that the $ mitigation is default-profile-only until resolved.

**Phase 3 — Telegram.**

**M19. getUpdates is at-least-once with implicit offset ack — replay/drop of destructive commands unhandled; 409-Conflict semantics unrecorded.** A bridge that crashes after executing a command but before its next poll re-receives and re-executes it on restart (duplicate /spawn; /kill against a since-respawned worker); ack-before-processing instead DROPS commands — nothing specifies where the offset persists or an idempotency rule, and two implementers pick opposite defaults. Telegram queues unconfirmed updates ~24 h, so a 6 h bridge outage replays a backlog of stale destructive commands against a fleet that has moved on — no staleness policy exists (OQ6 covers only the outbound direction). And a second concurrent getUpdates already receives HTTP 409 Conflict server-side — the correct primary dual-bridge guard, demoting OQ6's planned lock to a nicety. *Amend OQ6/constraints:* persist the getUpdates offset atomically in state/, ack-after-successful-dispatch with a per-update_id dedup journal (restarts neither drop nor double-execute); key confirmation round-trips to update_id so a redelivered confirmation cannot double-fire; max command age for mutating commands (discard older than N minutes with a summary reply; decide read-only exemption); treat 409 as another-bridge-running and exit loudly, lock file secondary.

**M20. Telegram auth "hard-checked against chat id" is underspecified across update types and identity fields; rejection disposition undefined; token-theft blast radius unaddressed.** Updates carry both chat.id and from.id — a from.id-only check executes commands the owner types in an attacker-created group. edited_message (owner edits → re-execution), channel_post, callback_query, and inline paths are unspecced — each a bypass or replay under a generic "get text, check id" loop. Rejection disposition (reply "unauthorized" = liveness oracle, vs silent drop, vs alert on repeated probes) is decided nowhere. Token theft from plaintext state/ lets an attacker concurrently call getUpdates (consuming the owner's queued inbound steering — full read) and impersonate the fleet outbound; OQ6 treats a revoked token only as availability. The reuse-existing-channel option (L20) has a richer access model the hard check must bind to. *Amend fixed constraints:* accept ONLY update type `message` where chat.type=='private' AND chat.id==owner_id AND from.id==owner_id; silently drop every other update type including edited_message; unauthorized traffic logged rate-limited to a state/ audit line, never answered; >N attempts/hour raises an owner notification; token file owner-only permissions (0600 via platform adapter, doctor-checked) + a compromise runbook line (revoke via BotFather; bridge detects 401 and alarms via the desktop notifier); the single-chat-id check binds the reuse path verbatim.

**M21. Reply-to threading has no defined behavior when the mapped worker is dead or cleaned.** The most common notification-reply path — replying hours later to a crash alert — maps message id → worker name → `fleet send`, which errors against a sticky-dead worker ("run fleet respawn first") or an unknown name after clean; no stub sentence says what the bridge does with the CLI error (reply "worker dead, /respawn?" vs swallow vs queue). OQ2 asks only where the message-id→worker map lives and its TTL. The phone-only done criterion guarantees this path is hit: crashed workers are exactly what generates phone notifications. *Amend OQ2:* reply-target validation — dead: offer inline /respawn confirmation, or queue to the old sid's mailbox so the respawn drain delivers it (compatible with journal-injection-at-respawn); cleaned: error reply with last-known result path. Must NOT auto-respawn (auto-recovery is watchtower's opt-in-only territory).

**M22. Two managers (phone on-call + live desktop session) steering one fleet have no arbitration or awareness model.** The mechanical layer is inherited (single-writer registry under fleet.lock; unique names make collisions deterministic errors), but: a phone /spawn puts a worker the desktop manager never spawned into its campaign (its wait/result collection silently misses it while status tallies include it); a phone /kill or on-call respawn can fire mid-`fleet wait`; contradictory sends from two intelligences merge into one mailbox unattributed. ROADMAP frames the on-call manager as THE chief-of-staff (singular) and the done criterion is phone-ONLY "while the desktop stays untouched" — coexistence is structurally out of the stub's field of view; none of OQ1–7 asks. *Amend:* add an open question with candidate dispositions: (a) act freely, document mailbox-merge and last-writer-wins for kill/respawn; (b) restrict destructive commands to confirmation round-trips (ties into OQ1); (c) share state via the durable chief-of-staff journal. Decide and name the carried invariants (single-writer registry untouched; one-live-claude enforced per-worker by existing guards).

**M23. The Telegram bridge is untestable without a real bot, and the single hard security property — chat-id auth gating the highest-privilege path — has no specified test.** The stub's only verification is one manual phone-only campaign, performed from the owner's (authorized) chat id — so the wrong-chat-id reject path can structurally never execute under the spec's own acceptance test. An auth bug means any Telegram user who discovers the bot can execute fleet commands on this machine, and nothing in the spec would ever have tested it. Reply-map TTL, 4096-char splitting, outage queue+flush, and the two-bridge guard are equally unverifiable short of live network chaos; long-polling via urllib is trivially fakeable. *Amend — add a Testing section:* all Bot API traffic through one transport function/class pointable at a local fake Bot API (stdlib http.server fixture with canned getUpdates/sendMessage); REQUIRED tests: wrong chat id → no fleet action AND no reply; command grammar parse table; reply-id→worker map incl. TTL expiry; >4096-char splitting; Telegram-down → outbound queued and flushed; second bridge refused; manager-on-call turn with a stubbed claude launcher (the test_steering.py fake-Popen pattern) asserting budget cap and cooldown. Phone-only campaign stays as acceptance soak.

**Phase 5 — Intelligence.**

**M24. The token-ceiling kernel has no enforcement lever compatible with the Stop hook's actual powers.** The stub fixes "per-worker cumulative TOKEN ceiling checked in the Stop hook" but never says what the hook DOES when exceeded. The Stop hook's only active lever is `{"decision":"block"}` — and block means the turn CONTINUES (SPEC L148), i.e. spends MORE tokens: blocking on over-ceiling is enforcement inverted. Hooks may not write registry/events (L78); cumulative-across-respawn counts live in registry+logs keyed by NAME while the hook has only session_id — so implementer A breaches the single-writer invariant while implementer B keeps the turn running. Two implementations, both wrong, from one sentence; neither the ceiling's config source nor the count's location is named. Not graveyard #3 (token-only, no price table, no PreToolUse). *Amend:* spec the lever explicitly — (1) over ceiling, the Stop hook ALLOWS stop even with mail pending (mail stays visible via idle+mail), never block; (2) hard enforcement lives in fleet.py launch paths (send-when-idle/respawn refuse or warn when cumulative tokens > ceiling), where registry access is legitimate; (3) the hook computes this turn's tokens from transcript_path in its stdin JSON (verify the field against the current hook contract), or spawn writes a sid-keyed ceiling file next to the mailbox; name exit-0 hooks and single-writer registry in the stub's invariants section.

**Cross-cutting.**

**M25. ROADMAP's mandated invariant checklist names only 5 of the 9 settled invariants; no stub carries an Invariants-touched section; Phase 6's multi-machine bullet already steers into the omitted ones with no stub to catch it.** ROADMAP L93 omits single-writer registry, one-live-claude-per-session, platform-adapter-only OS branching, and one-state-many-views — precisely the invariants Phases 2–4 stress hardest (watchtower writing registry/events; auto-respawn racing attach; web server invoking fleet code paths): a drafter dutifully checking the listed five misses exactly the collisions this review found. None of the seven stubs contains an invariants section. Phase 6's "registry/state sync via git or Tailscale-mounted state" (L84) violates the omitted set three ways: state/ is gitignored by the SPEC layout; a shared writable fleet.json means atomic-rename locking over a network filesystem (atomicity not guaranteed → single-writer broken); turn_pid/ctime liveness is machine-local, so machine B recomputes machine A's working worker as dead and `interrupt` can taskkill an unrelated local PID. *Amend:* expand ROADMAP L93 to the full nine — better, a numbered invariants section in SPEC.md referenced by number; add a mandatory "## Invariants touched" skeleton to every stub now; rewrite the Phase-6 bullet as per-machine registries with a read-only federation view — never a shared writable fleet.json or git-synced state/; liveness and interrupt stay machine-local.

## 6. Disposition appendix

### 6.1 Refuted findings (do not re-litigate)

1. **"events.jsonl single-writer is program-identity not process-identity; appends unlocked; 13-kind event vocabulary has no schema."** Refuted: the "appends are unlocked today" leg is false — every append_event call site runs under fleet_lock per the documented discipline (the one genuinely unlocked append, turn_commit_failed, is a best-effort forensics line in an already-critical path writing to a zero-consumer file); the ownership question is already tracked in the watchtower stub's L20 "(resolve: …)" marker, which states the very resolution proposed (single-writer = fleet.py process whoever it is, lock-guarded append); and the typed-event schema is explicitly scoped Phase-2 spec work ("typed events" in scope), with all consumers downstream of that spec. What survives is one-line editorial polish for the Phase-2 drafter, not a major finding.
2. **"`fleet status --delta` depends on the watchtower keeping the cursor, violating daemon-additive."** Refuted as a verbatim duplicate of an earlier-wave confirmed finding; the substance is retained here as a Phase-2 amendment note so nothing is lost: **cursors are plain per-consumer files in state/ written by whichever fleet.py invocation performs the delta read; the watchtower is just one more named consumer; --delta works identically with the watchtower dead; keep OQ6 for keying only.** The Phase-2 drafting session should treat that as a required constraint alongside M9–M16.
3. **"Journal-quality gate deadlocks a dead worker and ping-loops a live one."** Refuted: the gate is scoped to the opt-in auto-respawn path on a live worker at threshold — manual respawn is never bound by it (reader constraint, daemonless-baseline goal, wrong-by-definition rule); a dead worker triggers the separate crash rule → notification → manual respawn; and the ping-half rests on a false premise (send-to-dead fails loudly naming the recovery lever; it cannot silently queue). What survives is exactly OQ4's already-tracked open work — and is superseded by M10's cut-or-cap decision.
4. **"DoD enforcement described three incompatible ways (exit-2 vs mailbox veto vs OQ2)."** Refuted: "exit-2 pattern" is prior-art attribution (PRIOR-ART L20 expands the steal as our existing exit-0 mailbox-block channel); the inheritance chain (SPEC §7 + exit-0 invariant, "wrong by definition" rule) forecloses an exit-2 implementation; the genuinely open evaluation-locus split is exactly what OQ2 tracks. At most a one-line wording polish on the L17 parenthetical.
5. **"SPEC §4 'image is claude' is false for every real Windows worker (npm claude.cmd shim)."** Refuted on the target machine: `shutil.which('claude')` — what resolve_claude_executable calls — resolves to a native claude.exe, so the predicate holds as written; the code additionally carries an _ALIVE_IMAGE_NAMES set {claude, node, cmd} with the ctime match bearing the actual guarantee. A one-line spec-wording nit for an install shape absent here; hand to the Phase-1.5 drafter as input, not a finding.
6. **"Manager-on-call meaning drifts across docs; watchtower OQ8 smuggles the capability early; ownership unassigned."** Refuted: OQ8 already names its variant distinctly ("Manager wake-up"), flags the danger, and mandates guardrail speccing; the workflow forces the decision at Phase-2 spec time; Phase-5's exclusion parenthetical is clumsy but operatively unambiguous. The load-bearing residue (concurrent triggers, registry-entry claim protocol, privilege bounds) is already captured as confirmed blockers B4 and B5. Residue: editorial cleanup only.
