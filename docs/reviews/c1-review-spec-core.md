## 1. Verdict

The v2.1 amendment is **substantively strong on the descriptive findings** — B1's sticky-dead / sticky-attached / scan-past-`type:"system"` classification (F17), the M2 guard ladder (F18), the M1 cost model (F21), the M6 hook write boundary (F25), and the U7 quarantine row (F30) all match shipped `fleet.py` faithfully. **UL1 is honest and sound** (detection genuinely deferred with recorded evidence, conservative fallback well-bounded, recompute exemption + resume lock discipline consistent — verdict below). But the pass has **one internal-contradiction defect and one systemic labeling defect that both misstate shipped code**, and both land on the highest-trust surface a build session reads (§4 state machine + §12 "required regressions"). The core problem: the v2.1 body advertises itself as PURELY descriptive ("brings the spec TO the shipped code; no code change proposed") while several amendments (F20/F22/F24) prescribe **unbuilt** behavior in present-tense phrasing indistinguishable from the genuinely-shipped ones, and one amendment (F17 pre-claim) describes a residual the code has since FIXED — so §12's blanket "correct code passes" is false for at least two named regressions. A verify/build session inheriting these would either write regression tests that fail against correct code, or "fix" correct code back to the buggy spec. Not fit to hand to build until findings 1–3 are corrected. Three majors, zero blockers, no minors surfaced worth the noise.

## 2. Findings

**MAJOR 1 — §4 pre-claim "NEVER demoted / cleared by respawn, not by recompute" contradicts shipped code (recompute DOES demote a stale pre-claim to dead); the named §12 regression would break correct code.** — *SPEC §4 "Status recompute is scoped" pre-claim bullet + §11 reboot row + §12 `preclaim_never_demoted`.*

> §4: "**`working` + `turn_pid = None` = launch-in-flight (pre-claim)** — written under `fleet.lock` before `Popen` … and NEVER demoted. … Residual: a CLI killed between the pre-claim write and `Popen` leaves a pinned-`working` record with `turn_pid = None`; it is cleared by `respawn`, not by recompute."
> §12: "`preclaim_never_demoted` — `working`+`turn_pid=None` survives recompute unchanged."

Shipped `recompute_status` (fleet.py) time-bounds the pre-claim via `LAUNCH_CLAIM_MAX_AGE_SECONDS = 600` and DOES demote it to `dead` through recompute: `if current_status == "working" and pid is None: if _launch_claim_expired(last_activity_iso): return "dead"`. `recompute_worker` opts in by passing `last_activity_iso=record.get("last_activity")`, so the production path demotes a stale pre-claim after 10 minutes. The code comment is explicit that this is a deliberate evolution past the spec's stated residual: "*was 'Residual (documented, not fixed)' above*… once `last_activity` is older than `LAUNCH_CLAIM_MAX_AGE_SECONDS`, this demotes straight to 'dead', making it recoverable through every normal dead-worker path (`fleet kill`, `attach --force`, `respawn`, `clean`) instead of requiring manual registry surgery." So the spec's two load-bearing claims — "NEVER demoted" and "cleared by `respawn`, not by recompute" — are both false against shipped code, and the §11 reboot row ("launch-in-flight (`turn_pid=None`) stays working until respawn") inherits the same error for any reboot outage > 600 s.

*Why it's wrong:* this is the flagship B1 amendment (the whole point of which was fidelity to the shipped state machine), and it re-introduces a stale rule the code deliberately fixed. Worse, a builder implementing the named regression `preclaim_never_demoted` verbatim writes a test that **fails against correct code**, or "fixes" fleet.py to never demote — reintroducing the permanent-zombie `working`/`turn_pid=None` record that item 1c's zombie-escape-hatch exists to prevent. *Fix:* amend the §4 pre-claim bullet to the shipped rule — a pre-claim is not demoted by recompute WHILE FRESH, but a claim older than the launch-claim max-age (real launchers stamp the PID within seconds) demotes to `dead` and becomes recoverable via any dead-path lever, not only `respawn`; update the §11 reboot row identically; reword §12 `preclaim_never_demoted` to "a FRESH pre-claim survives recompute; a stale (> max-age) pre-claim demotes to dead," matching the regression the code actually pins.

**MAJOR 2 — §4's "three-way PID probe" (alive-unknown tier + retry-before-dead + Get-CimInstance fallback + §5 doctor elevation check) is UNBUILT; shipped probe is a two-way boolean and §12 `probe_three_way` will not pass.** — *SPEC §4 "PID liveness — three-way probe (v2.1, M4)" + §5 doctor row elevation-mismatch check + appendix F20 + §12 `probe_three_way`.*

> §4: "The stored `turn_pid`/`turn_pid_ctime` classifies into exactly one of three verdicts, never a bare alive/dead boolean … **alive-unknown is never demoted to `dead` …** any `working`→`dead` transition first retries the probe once after a short delay … The probe separates these via a try/catch marker in the PowerShell script (optional `Get-CimInstance` fallback)."
> F20: "Fix: the probe distinguishes alive-and-matching / definitely-gone / **alive-unknown** … retries once after a short delay before any `working`→`dead`; a same-elevation operating rule + doctor check are documented."

Shipped `pid_alive` returns a bare boolean (`return abs(...) <= 2.0`, else `False` on every failure branch), and `get_process_info` collapses EVERY failure mode — including the exact `$p.StartTime` `Win32Exception` the spec cites — into `return None` (`except Exception: return None`; empty stdout → `return None`). `recompute_status` then treats `not pid_alive(...)` as gone with no retry and no alive-unknown branch: `if pid_alive(...): return "working"; return "idle" if _last_line_type == "result" else "dead"`. Grep confirms fleet.py contains no `Get-CimInstance`, no `elevation`, no alive-unknown state, and no probe-retry anywhere. The M4 bug the review found is therefore **still present in shipped code**, and F20 describes the intended fix as though it were shipped.

*Why it's wrong:* §12's regression-suite preamble asserts "correct code passes, spec-as-written-in-v2 would fail" — but `probe_three_way` ("probe returns alive-and-matching / definitely-gone / alive-unknown; … a `working`→`dead` transition retries the probe once first") **fails against the shipped two-way probe**. A verify session trusting §4/§12 would believe the elevated-worker false-negative is closed when it is not (a live turn under an unreadable PID still recomputes to `dead`, the second-live-claude hazard M4 named). *Fix:* either (a) relabel F20/§4/§12 as PRESCRIPTIVE — an unbuilt fix owned by a hardening kernel (like the M6 PostCompact registration is honestly future-tensed in §7: "The template above **will** gain a THIRD entry") — with a `[UNBUILT]`/kernel-owner marker and removed from the "correct code passes" set; or (b) if it is meant to be shipped, that is a code gap the review should raise for C2, not a spec-fidelity claim.

**MAJOR 3 — the v2.1 header "Body brings the spec TO the shipped code; no code change proposed" and §12's "correct code passes" are globally false: F20/F22/F24 prescribe unbuilt code in present-tense phrasing a build session cannot distinguish from the genuinely-descriptive amendments.** — *SPEC v2.1 header line + §12 required-regressions preamble + appendix F22/F24 (and their own kernel cross-refs) + §4 schema block.*

> Header: "Body brings the spec TO the shipped code; no code change proposed."
> F22: "Fix: added `max_budget_usd` + `setting_sources` to the §4 schema (spawn-recorded, IMMUTABLE like `mode`); EVERY launch path re-passes them."
> F24: "rotation RETRIES briefly on `PermissionError` and, on continued failure, FAILS the respawn cleanly … with an `_unrotate` rollback … naming the likely holder."

None of these three is shipped. (a) **Budget/setting-sources (F22/M5):** `new_worker_record` — whose docstring claims "*matching the SPEC §4 schema exactly*" — omits `max_budget_usd` and `setting_sources` entirely; only `cmd_spawn`'s `launch_turn` call passes them (from CLI args), while the send-when-idle call (`launch_turn(... model=after.get("model") ...)`, no budget) and the respawn call (`launch_turn(... model=model ...)`, no budget) do NOT re-pass them, and nothing reads `record.get("max_budget_usd")` anywhere. The M5 "evaporates on turn 2" bug is intact. (b) **Rotation retry (F24/B2):** `_rotate_worker_log` is a bare `for suffix in (".jsonl", ".err"): os.replace(...)` with no try/except, no retry, no holder-naming; the call at respawn is unguarded and outside the launch try-block, so a `PermissionError` propagates as an uncaught traceback — not a clean-fail-with-rollback. (c) **Probe (F20):** see Finding 2. The amendments' OWN cross-refs concede the build is future work ("the `phase1-hardening-kernels.md` afternoon-kernel line is owned by `stub-inject-kernels-webui`"; UL1/F31 "the C2 build is hardening kernel item 11") — which directly contradicts the header's "no code change proposed."

*Why it's wrong:* the body silently mixes two amendment classes — **descriptive** (F17-core/F18/F21/F25/F30, verified to match code) and **prescriptive** (F20/F22/F24/F23/F31, unbuilt) — under a banner and a §12 preamble that assert only the descriptive kind. A build/verify session cannot tell which invariants are already enforced and tested versus which are TODO, so it will either skip building M4/M5/B2 (thinking them shipped) or grade a green-but-buggy binary as spec-conformant. This is the exact "inherit SPEC verbatim propagates known-wrong ground truth" hazard the review campaign exists to kill. *Fix:* correct the header to state the body folds BOTH already-shipped drift AND not-yet-built fixes; tag every prescriptive amendment (F20, F22, F23, F24, F31) with an explicit `[UNBUILT — owned by <kernel>]` marker matching the honest future-tense F25/§7 already uses; and split §12 into "regressions correct code passes TODAY" versus "regressions that pin unbuilt fixes (will fail until the kernel lands)," moving `probe_three_way` and the corrected `preclaim_never_demoted` accordingly.

## 3. UL1 verdict (extra-scrutiny lens 4)

**Clean.** The `limited` design does not resurrect or silently strand a worker: it is exempt from the `working`→`dead` crash path exactly as `dead`/`attached` are sticky (and, being neither `working`+non-None-pid, is untouched by shipped recompute scoping — consistent), its only exits are `now >= limit_reset_at` → `resume-limited` or `respawn --force`, and a null-horizon park is surfaced (`status` "reset unknown", `doctor` NOTE) rather than auto-resumed blind — no silent strand. Detection is honestly deferred: `claude --help` (2.1.204) was probed and recorded as exposing no usage-limit surface, the live wall is correctly noted as unforceable without burning real plan usage, and the exact signal is `DEFERRED-TO-KERNEL-PROBE` with a conservative fallback (errored + no `result` + limit-shaped stderr → `limited-suspected`, park + operator-confirm; non-limit-shaped errored stays crash-dead) — no invented signal presented as verified. `resume-limited` obeys one-live-claude / single-writer (pre-claim under `fleet_lock`, universal drain, re-passes spawn-immutable fields). The `limited`/`limit_reset_at`/`limit_kind`/`resume-limited` machinery is (correctly) absent from shipped `fleet.py` — it is C2 kernel-item-11 work, and the spec labels it as such. **One non-blocking caveat:** "limit-shaped stderr" is undefined (no pattern), so the fallback's crash-vs-park boundary is only as sound as the kernel probe makes it — acceptable because it is explicitly deferred, but the kernel MUST pin the pattern or the fallback can mis-park a genuine crash whose stderr coincidentally matches. (Note: UL1 escapes Finding 3 because it is honestly future-tensed and cross-refs its kernel — it is the MODEL the F20/F22/F24 amendments should have followed.)
