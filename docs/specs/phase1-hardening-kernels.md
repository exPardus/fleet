# Spec: Phase-1 hardening kernels (salvaged from idea-forge run 1)

**Status:** ready-for-build (amended v2) (pre-vetted — these are the judge-salvaged kernels from `docs/IDEA-FORGE-REPORT.md` §4; do NOT expand them into the dead parent ideas, see report §5 graveyard. v2 amendment 2026-07-08: kernel 3 hardened + items 6–11 folded from SPEC-REVIEW findings F27/F13/F9/F15/F12 and the UL1 usage-limit feature.)

Afternoon-sized line items for the Phase-1 build. Each is deliberately minimal; scope creep here is how the parent ideas died.

1. **Swallowed-hook exception log.** Hooks keep the exit-0-on-any-error invariant, but the except blocks append one line (timestamp, session_id, exception repr) to `state/hook-errors.log`. `fleet status` shows a total count when nonzero; `doctor` shows the tail. (~5 lines)
2. **Static hook-registration lint in `fleet doctor`.** Parse the rendered worker-settings.json: known event names, command paths exist, JSON schema shape. Catches typos the synthetic smoke test false-greens on. (small)
3. **Fail-silent PostCompact journal landmark.** PostCompact hook appends a mechanical line to the worker's journal: turn count, token count, timestamp, "context compacted here". Never blocks, never loud. (~10 lines)
   <!-- F11 --> **Amendment (F11 = review M6).** The kernel is under-specified as written: it names a hook but neither registers it nor pins the event name. Agreed fix: add the template/§7 registration, specify sid→name resolution with a sid-keyed fallback or skip, verify the actual event name against the installed CLI, and teach kernel 2's lint the new name. Concretely — (a) the hook is registered in the rendered `worker-settings.json` template AND documented in SPEC §7 alongside the other hooks; (b) the hook resolves the worker's journal by mapping `session_id`→worker name, and where that resolution fails it uses a **sid-keyed fallback path or skips** (never guesses a name, never writes to the wrong journal); (c) kernel 2's static hook-registration lint learns the new event name so a typo false-greens no longer. **BLOCKED-with-evidence rule applies at build time:** the C2 builder verifies the real PostCompact event name against the installed CLI; if the CLI exposes no such event, the builder reports BLOCKED with evidence and never builds against a guessed event name.
4. **Abnormal-turn-end journal note + cwd preflight.** When status classification finds a crashed turn (stale PID, no result), append "turn ended abnormally at <time>" to the journal so respawn context knows. Before any spawn/resume: stat that registered cwd still exists, clear error if not. (small)
5. **Spawn-time model echo.** Spawn output includes the effective model/config line (incl. `CLAUDE_CODE_SUBAGENT_MODEL` if set) so cost surprises are visible at launch. (one line)

<!-- item6-F27 --> 6. **Scripted live-integration harness (F27 = review M7).** A repeatable tier-3 harness — not the manual one-off smoke that found four bugs 338 green unit tests missed. Gated on `FLEET_LIVE=1`, uses the haiku model, a temp `FLEET_HOME` + a throwaway git repo, a hard **≤$0.25 budget ceiling**, and replays the documented smoke scenarios; each run captures sanitized stream-json logs into the fixture corpus (the `harness-live` C2 acceptance target). **Hook-source parameter (PLAN §0.1.6):** the harness renders the temp FLEET_HOME's hook paths from an explicit hook-source parameter (default = main-checkout hooks; override = worktree hooks) — the override is safe because it runs inside the temp-FLEET_HOME sandbox, and it is how changed hooks get pre-merge live execution before their worktree is merged. Graveyard-adjacency: a live harness is **not** exit-code capture of detached turns (report §5 graveyard) — it launches turns it owns and reads their stream-json, it does not reap orphaned detached PIDs.

<!-- item7-F13 --> 7. **Budget / setting-sources persistence (F13 = review M5).** `max_budget_usd` and `setting_sources` persisted in the registry and re-passed on every launch path (spawn, resume, respawn) under the M1 additive-schema rule; before any resume, a fleet-side cumulative-cost check runs so a cheap per-turn cap can't be circumvented by repeated resumes. The persisted values are re-passed on EVERY launch path — a missing re-pass is the failure mode this closes.
   **Witness:** "max_budget_usd and setting_sources persisted in the registry and re-passed on every launch path"

<!-- item8-F9 --> 8. **Send-lock serialization + orphaned-mailbox doctor + mail events (F9 = review M3).** `send`'s append to the working/attached worker's `mailbox\<sid>.md` happens under `fleet_lock` (or re-reads the sid and re-appends), and respawn's drain+sid-swap is atomic with respect to `send` so no message lands in the pre-swap mailbox after the swap. `doctor` gains an "orphaned mailbox" definition that **extends the existing orphaned-claim check** rather than duplicating it. The single writer also emits `mail_sent`/`mail_drained` events with fleet.py as the sole writer — the Soak-1 audit enabler, not a DLQ. Graveyard-adjacency: these events are **not** a mailbox dead-letter queue (report §5 graveyard) — no retry, no reraise, no redelivery machinery; they are append-only audit records under the single-writer registry invariant, answering graveyard #8's cause of death.
   **Witness:** "mail_sent/mail_drained events with fleet.py as the sole writer — the Soak-1 audit enabler, not a DLQ"

<!-- item9-F15 --> 9. **PID-probe failure semantics (F15 = review M4).** PID probe is three-way — alive-and-matching / definitely-gone / exists-but-unreadable = alive-unknown never demoted to dead — with a retry-once-before-dead step to absorb a transient read, plus a `doctor` check for an unreadable `StartTime`. An unreadable-StartTime process is treated as alive-unknown, never silently classified dead, so a live worker is never reaped on a probe hiccup.
   **Witness:** "PID probe three-way — alive-and-matching / definitely-gone / exists-but-unreadable = alive-unknown never demoted to dead"

<!-- item10-F12 --> 10. **Token-ceiling kernel (F12 = review M24; F14 = review M18).** A per-worker token ceiling distinct from the dollar cap: over ceiling, the Stop hook **ALLOWS stop even with mail pending, never block** (the hook can only ever exit 0 / allow — invariant 2), and hard enforcement lives in fleet.py launch paths which refuse to launch a turn that would exceed the ceiling. The ceiling is read from a sid-keyed ceiling file or computed from `transcript_path` token accounting. Graveyard-adjacency: the ceiling is **not** a per-invocation dollar table — it is a token count enforced fleet-side; the Stop hook never gains blocking power from it.
    **Witness:** "over ceiling, the Stop hook ALLOWS stop even with mail pending, never block; hard enforcement lives in fleet.py launch paths"

<!-- item11-UL1 --> 11. **Usage-limit resilience kernel (UL1 = NEW FEATURE, no review finding).** Binding input: `docs/reviews/USAGE-LIMIT-RESILIENCE-INTENT-2026-07-08.md`. When a `claude -p` turn ends at a Claude PLAN usage-limit wall (5-hour rolling session OR weekly cap) — not a crash, not a finished task, not a per-worker budget-trip — fleet parks a worker that hit a Claude plan usage limit in a `limited` state with a recorded `limit_reset_at`, and auto-resumes through the `fleet_lock`-guarded launch path once the window resets. Detection is env-dependent → **BLOCKED-with-evidence / live-probe at build time:** the C2 harness (item 6) confirms the real signal against the installed CLI before anything is built against it; no signal is invented, and until confirmed the conservative `limited-suspected` fallback governs (park + surface for operator confirmation). This kernel implements the contract specced by the SPEC amendment (link `spec-amend-4-usagelimit`, appendix F31), demonstrated with a live-harness run whose hook-source follows §0.1.6. Graveyard-adjacency: parking + explicit resume is **not** a dead-letter queue or auto-adoption — the worker's own mailbox+journal are preserved and resume runs the ordinary lock-guarded launch path (invariants 6, 7), no redelivery/retry machinery.
    **Witness:** "fleet parks a worker that hit a Claude plan usage limit in a `limited` state with a recorded limit_reset_at, and auto-resumes through the fleet_lock-guarded launch path once the window resets"

## Invariants touched

Cites the numbered "Architectural invariants (numbered)" section of `docs/SPEC.md`.

- **exit-0 hooks (2)** — the PostCompact landmark (kernel 3) and the token-ceiling Stop hook (item 10) both exit 0 unconditionally; the Stop hook can only ALLOW, never block. Hooks never gain enforcement power; enforcement lives fleet-side.
- **single-writer registry (6)** — items 7, 8, 10, 11 add fields/events but `fleet.py` remains the sole writer of `fleet.json` under `fleet.lock`; the `mail_sent`/`mail_drained` events are single-writer audit records, not a second writer.
- **one-live-claude-per-session (7)** — items 8 (send/respawn atomicity), 9 (never reap a live worker on a probe hiccup), and 11 (resume through the pre-claim launch path) each preserve the one-live-claude guarantee.

Graveyard-adjacency answered inline per item: harness ≠ exit-code capture (item 6); ceiling ≠ $-table (item 10); drain-event ≠ DLQ, no retry/reraise machinery, single-writer preserved (item 8).

Explicitly rejected even as kernels (do not build): mailbox dead-letter queues, PreToolUse scope guards, amnesia/journal-read verification, orphan auto-adoption, hook reraise/fail-loud, exit-code capture of detached turns.
