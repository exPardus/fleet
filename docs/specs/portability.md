## Goal

| **D2** | **Rewritten — fix-wave F1 (CRITICAL).** Linux liveness probe: read `/proc/<pid>/stat`. Parse `comm` by finding the **last** `)` in the line (`comm` is unquoted and may itself contain `)` or spaces — inferred from the field's free-form nature, not a verbatim man-page quote; see the evidence cell's F16 correction), then split the remainder on whitespace: index 0 of that remainder = field 3 (`state`, first character only — see D4/F12); `starttime` (field 22) is index 19. **Do NOT recompute wall-clock via `time.time() − /proc/uptime` on every read** — that mixes CLOCK_REALTIME (NTP-steppable) with a monotonic quantity and was the CRITICAL bug: the stored `turn_pid_ctime` is produced by the same formula at launch, so any REALTIME step of `S` seconds after launch shifts every later probe's derived value by `S` while the kernel's `starttime` field never moves, so the ctime compare falsely reads `abs(diff) > 2s` → `"gone"` → dead → `respawn` double-launches in the worker's cwd (invariants 5, 7 — the same double-launch class C2's review already caught once on Windows). **Fix: treat the raw `starttime` ticks as a synthetic, monotonic anchor instead of recomputing a real wall-clock instant:** `ctime = datetime(1970, 1, 1, tzinfo=timezone.utc) + timedelta(seconds=starttime_ticks / os.sysconf('SC_CLK_TCK'))`. (b) **Reboot** — boot-relative ticks reset near-zero for every process after a reboot; a `turn_pid_ctime` stored pre-reboot differs from any freshly-probed post-reboot process's ticks by (order of magnitude = pre-reboot uptime-at-launch) seconds, far outside ±2 s, so a genuinely-rebooted worker correctly classifies `gone` (matches SPEC §11's reboot row; this is the *safe* direction — the process really is gone). ### POSIX attach argv, pinned (F9)

The Windows side pins argv byte-for-byte (`["wt", "-d", cwd, "--", "claude", "--resume", sid]`) and its tests assert exact equality; the earlier draft specified only a priority *order* for POSIX with zero argv, which left `port-test-suite` unable to write the four mandated tests without inventing the contract. Pinned here, in priority order:

- **`$TERMINAL` env branch** (checked first — `os.environ.get("TERMINAL")`, used as-is, not `which()`-resolved: the operator set it deliberately): `[$TERMINAL, "-e", "sh", "-c", f"cd {shlex.quote(cwd)} && claude --resume {shlex.quote(sid)}"]`. This is the ONE shell-mediated branch: terminal emulators' `-e` conventions diverge (`-e cmd args...` vs `-e "single string"`), so routing through an explicit `sh -c "<string>"` argv element is the one convention portable across `-e`-supporting emulators. It is still a single argv *list* handed to `Popen` — `sh` is a visible, explicit child argv element, never `Popen(..., shell=True)` — so this does not contradict the "argv list never goes through a shell" framing used elsewhere in this spec (that framing describes the `Popen` call itself, not whether an argv element happens to be a shell).
- **`which("gnome-terminal")` branch** (second): `["gnome-terminal", f"--working-directory={cwd}", "--", "claude", "--resume", sid]` — gnome-terminal's own documented `--` separator; no shell, matches the Windows `wt` pattern exactly.
- **`which("tmux")` branch** (third, ONLY when already inside a tmux session — detected via `os.environ.get("TMUX")` being non-empty; if `tmux` is on PATH but `$TMUX` is unset, this branch is SKIPPED, not used to start a brand-new detached tmux server): `["tmux", "new-window", "-c", cwd, "claude", "--resume", sid]` — opens a new window in the CURRENT tmux session/server. No shell.
- **Headless fallback** (nothing above matched): `None` — see the consumer-side note.

`$TERMINAL` wins if set even when `gnome-terminal`/`tmux` are also available, on the theory that an operator who set `$TERMINAL` picked it deliberately.

**Consumer-side note (F8 fix — state transition specified):** `cmd_attach` (`bin/fleet.py:3156`) commits the `attached` pre-claim FIRST, under the lock (`bin/fleet.py:3296-3301`), BEFORE it ever calls `PLATFORM.build_attach_argv` (`:3302`) or `popen` (`:3304`). The existing `except BaseException: ... roll back to idle` block (`:3305-3314`) exists only to undo the pre-claim if `Popen` itself throws — the headless print path must NEVER enter that block, because nothing needs undoing. **Decision: the `None`-check sits after the pre-claim and before the `popen` call; on `None`, `cmd_attach` prints the `claude --resume <sid>` command and the cwd, and returns WITHOUT calling `popen` and WITHOUT rolling the status back — the worker stays `attached`.** This is the safe direction: `recompute_status` already refuses to demote `attached` (`bin/fleet.py:~789`), so a worker whose operator never runs `fleet release` is pinned `attached` forever with no live process — safe, if inconvenient, and it must be *chosen*, not left to a builder's guess. The alternative (rolling back to `idle`) would let a live interactive `claude --resume` started by hand from the printed command race a subsequent `fleet send`'s resume launch into the SAME cwd — two live `claude` processes, one cwd, invariants 5 and 7. **The printed output MUST instruct the operator that `fleet release` is required when done** — this is the operator-facing cost of the safe direction and must appear in the printed message itself, not just in this spec. `port-posix-smoke`'s done-criterion-6 observation of the headless attach path must confirm both the printed instruction and that the worker record reads `attached` (not `idle`) immediately after. This `None`-check is a generic return-value branch, not an `os.name`/`sys.platform` check, so it does not trip `TestPlatformAdapterBoundary`'s source-scan and does not weaken invariant 8 — the *decision* of which launcher to use (or none) still lives entirely inside the adapter.

**`UnsupportedPlatformError`** (`bin/fleet.py:204-206`) stays defined (nothing deletes it — `test_unsupported_platform_error_is_not_implemented_error` still needs the class to exist) but is no longer raised by any `_PosixPlatform` method once this ships; it becomes dead-but-harmless unless a future OS adapter is stubbed the same way.
