# Fleet Self-Build Implementation Plan — Remaining Roadmap, Risk-First (Consolidated)

**Contract version:** 1.3 (round-1 plan review + completeness-critic pass applied) — 2026-07-08 · branch `fleet-impl` @ 792583c · executed by a fleet manager session per `skill/SKILL.md` doctrine. v1.3 adds §0.6 progress ledger (`docs/PLAN-PROGRESS.md`) + runbook steps 2/11 for cross-session continuity — the one gap the completeness critic surfaced.

**Angle (committed):** RISK-FIRST. The biggest blowup modes are (1) building against a stale spec, (2) the fleet mutating `bin/fleet.py`/hooks while workers run on them, (3) 338 green unit tests coexisting with live bugs, (4) Windows-pinned assumptions leaking into every later phase. So the order is: **spec amendments → self-build safety protocol + scripted live-integration harness → hardening kernels → only then feature phases.** We accept slower visible progress; nothing feature-shaped ships until the fleet has proven it can safely rebuild itself. New in v1.1: demand-gating pressure valves (providers, web-UI build, intelligence build) so a one-human part-time operator is never building unfelt features; read-only Web UI v1 and flag-sized Phase 5 cut the back half's cost roughly in half without touching the roadmap's done-criteria. New in v1.2 (round-1 review): harness hook-source override so hook changes get pre-merge live verification; headless permission mechanisms replacing unusable bare `accept`; the finding corpus (`docs/reviews/SPEC-REVIEW-2026-07-08.md`) committed and made the mandatory quoted input; merge gate reordered with an explicit revert path; per-link pytest truth checks; timed cost-watch polls with honest residual-risk statement; recomputed budget envelopes; witness-sentence verification + split C1 review; defined, non-circular soak audits with fault injection and usage-denominated gates.

## 0. Global operating protocol (applies to every campaign)

### 0.1 Self-build safety protocol (bootstrap hazard: fleet modifying fleet)

The manager and all workers execute via the **live** fleet install at `C:\proga\claude-fleet` (registry, hooks resolved through `state\worker-settings.json` → `C:/proga/claude-fleet/bin/hooks/*`). Therefore:

1. **Code-touching workers never work in `C:\proga\claude-fleet` itself.** The manager creates one git worktree per code campaign: `git worktree add C:\proga\claude-fleet-wt\<campaign> -b <campaign>` and spawns code workers with `--dir C:\proga\claude-fleet-wt\<campaign>`. Edits to the worktree's `bin/fleet.py` and `bin/hooks/*` cannot hot-swap the live CLI or the live hooks (hooks are absolute-pathed to the main checkout). This is the version-pin.
2. **Doc/spec-only workers may run in the main repo** (`--dir C:\proga\claude-fleet`) on strictly disjoint files, staging by exact path with index.lock retry (Campaign-0 lesson: disjoint-file parallelism is safe; same-file is not).
3. **Merge gate (campaign boundary only) — pre-merge checks first, explicit red-path after (revised):**
   - **Pre-merge, on the worktree branch:** `fleet status` shows **zero turns in flight** → `py -3.13 -m pytest tests/` (full unit+hook suite) green **in the worktree** → additive-schema diff check (§0.1.5) → **spec-vs-code drift check**: manager diffs the campaign's code changes against SPEC/stub sections and confirms every task's RESULT `spec=` field (§0.2.8) is consistent with the diff. Any red = no merge; fix task in the worktree.
   - **Merge** worktree branch → `fleet-impl`, then immediately — and with **no fleet command other than the gate checks between merge and green** — run in order: `FLEET_LIVE=1` integration tier (once it exists, C2 onward) → `fleet doctor` → `fleet init` re-run iff `worker-settings.template.json` changed → post-merge hook-smoke (§0.1.4) if hooks/template touched.
   - **Red path (explicit):** any post-merge red = the manager's **next action is `git revert` of the merge commit on `fleet-impl`** (restoring the known-good live install), then a fix task in the worktree. Never "fix forward" on the live branch, and never dispatch any worker between a red gate check and the revert.
4. **Post-merge hook-smoke (standing rule):** after **any** merge that touched `bin/hooks/*` or `worker-settings.template.json`, the manager spawns one throwaway haiku worker in a temp dir (budget cap $0.50) and confirms via its journal/mailbox artifacts that **both** hook events fired on the new hooks **before dispatching any real worker**. This is the post-merge backstop; it is no longer the *only* live check on changed hooks (see §0.1.6).
5. **Additive-only schema during campaigns (standing merge condition):** registry/`state/fleet.json` schema changes must be additive — readers default missing fields, writers preserve unknown fields — and **no migration that breaks the live `state/fleet.json` mid-campaign may merge**. This is a merge-gate check (manager inspects the diff for schema-touching code), not merely SPEC text.
6. **Iron rule (from amended SPEC §12) + hook-source override (revised):** no change to `launch_turn`, `bin/hooks/*`, or stream parsers merges without a green `FLEET_LIVE=1` run **that actually executes the changed code**. The harness (C2 `harness-live`) therefore takes an explicit **hook-source parameter**: default = **main-checkout hooks** (used for merge-gate runs and any run where hooks are not under edit); override = **worktree hooks**, rendering the temp FLEET_HOME's `worker-settings.json` hook paths to the worktree's `bin/hooks/*`. The override is safe: the temp FLEET_HOME sandbox isolates the run from the live install — the §0.1 hazard was only ever the LIVE registry/hooks, never a sandboxed copy. **Pre-merge live verification of any `bin/hooks/*` change MUST run with hook-source=worktree** (see 2A-close gate and `harden-fleet-d`); the post-merge hook-smoke (§0.1.4) plus the revert path (§0.1.3) remain the backstop. Until C2 delivers the harness, `launch_turn`/hook/parser changes get a manual haiku smoke (budget ≤ $0.25) replaying `.superpowers/sdd/integration-smoke.md` scenarios.
7. **Same-file sequencing, fleet-wide:** at most **one** worker with `bin/fleet.py` in its write set is alive at any moment, across all campaigns. `tests/*`, `bin/hooks/*`, `docs/*` are separate write sets and may run in parallel with it.
8. **Interim cost-watch (revised; active from C1 until `harden-fleet-b` merges):** budget/setting-sources persistence does not yet survive past a worker's first turn, so during this window the manager runs a **timed poll loop**, not a blocking wait: `fleet wait <names...> --all --timeout 300` in a loop (`wait` takes positional names — there is no bare `--all` form; timeout expiry exits non-zero and wakes the manager). On **every wake (≤5-minute worst case)**: `fleet status` — kill any worker whose `cost_usd` exceeds **2× its stated cap** (tasks without an explicit cap use the §0.4 class default — the kill threshold always has a number); AND `fleet peek <name>` on each `working` worker — kill on mid-turn anomalies (cumulative token counts or turn duration wildly out of proportion to the task). **Honest residual risk, stated:** `cost_usd` is computed only from *completed* turn result events, so a single runaway resume turn shows $0 until it ends — the poll bounds detection latency for *multi-turn* runaways to the poll interval and gives only the peek proxies (tokens/duration) mid-turn; there is **no hard cap** on a resume turn until `harden-fleet-b` merges. This is why `harden-fleet-b` sits at position 2 of the 2B chain — only `harden-fleet-a` (the small harden-hooks interface counterpart) precedes it, an exposure window of one link (~1–2 days) under the poll rule; that trade is accepted and recorded here rather than presented as a functioning cap.
9. **Chain-link truth gate (new standing rule):** before dispatching the next link of **any** `fleet.py` chain (2B, C4/C5/C5b/C6/C7/C8 chains), the manager personally runs `py -3.13 -m pytest tests/` in the worktree and confirms the finished link's RESULT-line pass count matches the actual run. Mismatch = the link is **not done**; the discrepancy goes back to the original builder via `fleet send`. The suite runs in ~8s — a per-link truth check is free compared to a same-file fix wave discovered three links later (Campaign-0's documented bottleneck).

### 0.2 Task-file convention (replaces prose boilerplate; every task, every campaign)

Every `--task` is written to `state/tasks/<name>.md` (runtime dir, gitignored) with this checkable structure:

1. **Precondition line (soak-gated campaigns only):** the first line of every next-phase spec/build task file is: *"Verify the previous soak gate's Altai signature line exists in `knowledge/lessons.md` (format: `SOAK GATE <n> SIGNED: <date> — Altai`); if absent, output `BLOCKED: missing soak sign-off` and stop."* This converts soak gates from manager memory into a machine-checkable precondition.
2. **Goal** — one paragraph.
3. **Exact paths owned** (the write set) and named function/line anchors — do NOT read `bin/fleet.py` end-to-end (2981 lines); Grep the named anchors and read only those regions.
4. **Quoted authority text (revised):** verbatim excerpts of (a) the SPEC/stub sections the task builds against AND (b) for every finding the task folds or implements, **the full finding text + agreed fix quoted from `docs/reviews/SPEC-REVIEW-2026-07-08.md`** — never this plan's one-line parentheticals (Campaign-0 lesson: ONE binding fix list, no re-interpretation). Note on numbering: this plan's B1–B5/F6–F30 IDs map to that document's B/M/U-numbered items — **the manager embeds the mapping (plan ID → review-doc item ID + quoted text) in each task file**; workers never grep for a plan-numbered ID (SPEC.md's appendix carries a *different, settled* F1–F16 series).
5. **Invariants touched** — which of the 9 named invariants, and why preserved.
6. **Verbatim done-criteria** (copied from this plan's task table).
7. **Repo rules block:** `py -3.13`; `bin/fleet.py` stdlib-only single file; forward slashes in hook commands; never Git-Bash `&`; runtime dirs gitignored, `knowledge/` tracked; pytest for unit/hook tests; test-first (paste failing-then-passing output in journal); commit early/small with exact-path staging + index.lock retry; commit-prefix rule for the campaign.
8. **Required RESULT contract (revised):** the final result message MUST end with a single line `RESULT: files=<changed paths> tests=<N passed>/<M total> spec=<SPEC/stub §-ref updated | no-drift> criteria=<one-line proof per done-criterion>` plus `git log --oneline -3`. The `spec=` field is mandatory for every build task — it is how R5's anti-drift clause is mechanically inherited by every task file, and the merge gate's drift check (§0.1.3) audits it. The manager verifies from `fleet result` only for *format and dispatch decisions*; a malformed RESULT line = task not done. **Truth of the `tests=` claim is verified per §0.1.9** (chain links) and §0.3(g) (phase close) — the RESULT line is a worker assertion, never itself the gate.
9. **BLOCKED-with-evidence rule (environment-dependent kernels):** if a task depends on a probed environment contract (e.g. the installed CLI's PostCompact hook event) and the probe fails, report `BLOCKED:` with the probe evidence — never build against a guessed contract.
10. **Permission mechanism line (new):** every task file states which of the three §0.4 mechanisms it runs under (`bypass`, `bypass-with-containment`, or `accept+allowlist`) and, for `accept+allowlist`, quotes the exact `permissions.allow` entries the manager provisions before spawn. A task whose core job is running commands may never ship under bare `accept` — headless print mode auto-denies un-allowlisted Bash/network/MCP.

### 0.3 Quality-gate pipeline (instantiated per phase; task letters referenced below)

(a) **Spec task** — claim stub (`Status: drafting (<date>)`, commit immediately), answer *every* open question (mark resolved, delete none), add mandatory `## Invariants touched` section naming which of the 9 invariants are touched and why preserved, check the IDEA-FORGE §5 graveyard for every addition. → (b) **Adversarial spec review** — independent worker attacks the draft; findings with exact quotes; ends with disposition appendix; spec worker (or fix worker) applies; `Status: ready-for-build` only after disposition. → (c) **Build tasks** — test-first, worktree, chained if same-file, per-link truth gate (§0.1.9). → (d) **Code review** — independent worker reviews the full campaign diff for spec conformance. → (e) **Adversarial break review** — separate worker actively tries to break the increment: lock races, kill-mid-launch, spaced paths, CRLF, handle-in-use on Windows, hook crash paths. Runs BOTH lenses per Campaign-0 lesson. → (f) **Adjudication + fix waves** — manager (or a dedicated adjudicator task) merges (d)+(e) into ONE binding fix list anchored to function roles, not line numbers. **Fixes go to the ORIGINAL builder via `fleet send`** (respawn first if past ~30 turns) — this keeps the builder's context and avoids the re-interpretation failure mode Campaign 0 documented; a fresh fixer is used only if the builder's session is dead/cleaned. **Re-review after every fix wave** (each wave introduces ~1 new issue); max 3 waves, then escalate to Altai. → (g) **Verification checkpoint — W-V discipline (doctrine):** the manager **personally re-runs** `pytest`, `fleet doctor`, and each done-criterion, with evidence pasted into the campaign log; a phase never closes on worker say-so. → (h) **Knowledge loop** — a `*-knowledge` task appends to `knowledge/lessons.md`, updates `knowledge/INDEX.md` (+ `knowledge/projects/` as relevant), commits. **Anti-ritual check:** the manager's verification of each `*-knowledge` task requires the lessons entry to name **at least one concrete process change** ("what to do differently" — normally an amendment to `knowledge/playbooks/campaign-template.md`); a lessons entry with no process change is rejected. Soak sign-offs land here as dated `SOAK GATE <n> SIGNED` lines.

### 0.4 Defaults

- **Model:** default model for build/spec/fix; strongest available for adversarial reviews (b/e); haiku only inside the live-integration harness's spawned test workers and hook-smoke workers.
- **Permission mechanisms (revised — replaces the old bare-`accept` rule):** fleet workers are detached headless `claude -p` sessions; `--permission-mode acceptEdits` auto-approves *file edits only*, and print mode **auto-denies** un-allowlisted Bash/network/MCP (SPEC §mode table) — there is no human at a prompt. So there are exactly three sanctioned mechanisms, justified per task (§0.2.10):
  - **`bypass`** — trusted grind in this known repo.
  - **`bypass-with-containment`** — for tasks that must run arbitrary live processes / kill PIDs / spend money where an allowlist is impractical: bypass mode PLUS written containment in the task file (temp FLEET_HOME, throwaway repo, worktree isolation, per-run budget ceiling, kill-targets restricted to harness-spawned PIDs). This is the mechanism `harden-fleet-d` already used; `harness-live`, `watch-accept-tests`, and `prov-live-demo` now use it too — identical live spend gets identical treatment.
  - **`accept+allowlist`** — mode `accept` PLUS a manager-provisioned scoped `permissions.allow` block written to `<worktree>/.claude/settings.json` **before spawn** (an explicit manager step; the exact entries are quoted in the task file and the settings file is excluded from the merge). Used where network/secrets argue for narrow capability: `port-ci` (`Bash(git push*)`, `Bash(gh *)`, `Bash(py -3.13 -m pytest*)`), `tg-live` (`Bash(curl https://api.telegram.org*)`, `Bash(py -3.13*)`), `web-live-check` (`mcp__plugin_playwright_playwright__*`, `Bash(py -3.13*)`).
  - `plan` first turn for genuinely unfamiliar territory, as before.
- **Budgets (revised):** every task carries `--max-budget-usd`; treat as circuit breaker (~3× overshoot possible, spawn-etiquette lesson), so caps are set at ~⅓ of true tolerance. **Class defaults for any task not given an explicit cap in its table:** review $10, adversarial review $12, knowledge $5, fix task $8, re-review $10, smoke/checklist task $4–6, `*-ci-fix` $6. These defaults are what the §0.1.8 2×-cap kill rule keys on — no task is ever dispatched without a number. **Iteration loops are capped in count as well as dollars** (e.g. `port-ci-fix-N`: $6 each, max 5, then Altai escalation per touchpoint 8). Campaign envelopes presented to Altai are **sum-of-caps including maximum fix/re-review loops** (the true dispatchable ceiling), with expected spend noted separately — the approval number is never below what the plan can dispatch. Interim cost-watch rule (§0.1.8) applies until budget persistence lands.
- **Respawn doctrine:** any worker past ~30 turns or confused → `fleet respawn`; journals make it lossless.
- **Demand-check pressure valves:** before the **providers build (C5b)** and before **each of the C7/C8 build waves**, the manager asks Altai one question: *"Do you feel the friction this phase removes, today?"* **The demand check gates the BUILD only.** The spec task (and its review) always runs on schedule — specs are cheap and perishable-context work — so a later "yes" finds a ready-for-build spec; **only the build waves park in the backlog** if there is no felt demand. Recorded as a dated line in lessons.md either way. This is the part-time-human realism lever: nothing before Phase 6 is built on schedule-momentum alone. (C5b's campaign text mirrors this rule verbatim — one rule, both places.)

### 0.6 Progress ledger (cross-session continuity)

4. **Worker journals (`state/journals/<name>.md`) + `fleet status`** — live in-flight worker state, for a session resuming mid-wave.
