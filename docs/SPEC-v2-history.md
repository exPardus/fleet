## 1. Problem & goals

One "manager" Claude Code session must spawn, monitor, steer, and collect results from multiple "worker" Claude Code sessions across arbitrary project directories on this machine. The human must be able to drop into any worker interactively at any time and hand it back. The manager acts as a general dispatcher ("another me"): parallel fan-out, long-running babysitting, cross-project routing, review pipelines — with a persistent, growing knowledge base.

**Non-goals (v1):**
- No daemon, no server, no background supervisor process.
- No true mid-turn conversational back-and-forth with a worker (mailbox injection at tool boundaries is the approximation).
- No cross-machine orchestration; no scheduling (manager does that itself).

## 2. Core design decisions

**Sessions, not processes.** A Claude Code session is durable state on disk addressable by `--session-id` / `--resume`. A worker is a *session*; each turn is a short-lived `claude -p` process. Crash-safe by construction. The registry tracks metadata plus the currently-running turn PID.

**Resume is cwd-scoped (load-bearing invariant).** `--resume <sid>` only finds the session from the same project directory (or its worktrees). Every turn launch and every attach MUST run with the worker's registered `cwd`. The `cwd` field in the registry is immutable after spawn.

**Why not the built-in `claude --bg` / `claude agents`?** CLI 2.1.202 ships background agents (`claude --bg`, `claude agents --json`). They cover spawn+list+monitor but provide none of: named registry with per-task permission modes, mailbox steering mid-turn, journals + respawn continuity, attach/headless conflict guards, or a knowledge loop. Fleet v1 uses its own detached launcher because the interaction of `--bg` with `--settings`-injected hooks and `--resume` turn cycling is undocumented; `fleet doctor` prints a note if `claude agents --json` shows fleet-unknown sessions. Revisit adopting `--bg` as the launch mechanism when its semantics are documented. **[Update, native-substrate pivot 2026-07-13]: revisited and adopted — fleet now dispatches onto exactly this substrate (native `--bg`), keeping only the semantic layer on top. See `docs/superpowers/specs/2026-07-13-native-agents-pivot-design.md` / `docs/specs/native-substrate.md`.**

## Architectural invariants (numbered)

The nine load-bearing invariants of the system. Later specs and every phase stub cite these BY NUMBER ("invariant N") — the wording here is canonical, and any phase whose work touches one carries an `## Invariants touched` section naming it (the ROADMAP L93 expansion to all nine and the per-stub skeleton are owned by Wave-1B stubs; this numbered list is the source they reference).

1. **daemonless launch** — no resident process is required; every action is a short-lived CLI invocation (§2). A watchtower/web-UI, if built, is additive: the CLI must work fully with it stopped.
2. **exit-0 hooks** — hook scripts always exit 0, never surfacing a traceback into the worker transcript (§7).
3. **atomic single-file mailbox** — steering is an append-only `mailbox\<sid>.md`, claimed by an `os.replace` rename (§7).
4. **journal-injection-at-respawn** — respawn rebuilds worker context from the journal; zero lost state (§8).
5. **cwd-scoped resume** — a worker resumes only in its recorded, immutable `cwd` (§2).
6. **single-writer registry** — only `fleet.py` writes `fleet.json`, guarded by `fleet.lock` (§4).
7. **one-live-claude-per-session** — never two live claude processes on one `session_id` (§6/§9).
8. **platform-adapter-only OS branching** — all OS-specific behavior sits behind the platform adapter; no scattered `if platform` in core (§14).
9. **one-state-many-views** — the registry + logs are the single source of truth; `status`/`peek`/`result`/web/telegram are DERIVED views, never independent state (§5).

**Hook write boundary (v2.1, M6).** Hooks NEVER write the registry (`fleet.json`) or the event log (`events.jsonl`) — those stay single-writer to `fleet.py`. Hooks MAY write, and only, these data-plane files: (a) `mailbox\<sid>.md` and its `.claimed.<pid>` rename; (b) `state\hook-errors.log` — append-mode, single-line-per-error, best-effort, NO lock (a multi-writer append file; NTFS single-line `open(...,"a")` writes are atomic enough for diagnostics; a lost line under contention is acceptable, never a crash); (c) their OWN worker's journal `state\journals\<name>.md`, and only via the PostCompact hook; (d) **(v2.3, M-B T2, native-substrate pivot)** their OWN worker's terminal-outcome record `state\outcomes\<name-or-sid>.jsonl` — append-only, via the Stop hook only, same NO-lock/best-effort discipline as (b) but with a single-syscall atomic append (Win32 `FILE_APPEND_DATA`) since a fleet-side tombstone writer may append to the same file concurrently. Registry access from a hook is READ-ONLY and failure-tolerant: a hook may read `fleet.json` to resolve its `session_id`→`name` (PostCompact needs this — journals are keyed by NAME but a hook receives only `session_id`), but on ANY failure (missing/locked/corrupt registry, sid not found) it falls back to a sid-keyed path or skips silently. **Every hook exits 0 on every failure** (§7) — the boundary rewrite does not weaken the exit-0 invariant.

**Cost model (v2.1, M1).** Per-invocation result costs are NOT cumulative: with `--resume`, each turn's `result` event reports the cost of THAT turn only, so a later turn can report a SMALLER number than an earlier one (SMOKE-B live-proved). A naive last-result-wins reader therefore undercounts. `cost_usd` is the lifetime spend of the NAMED worker, computed as `cost_baseline + Σ(result-event costs in the CURRENT log)`. `cost_baseline` is stamped by `respawn` when it rotates the log (= the pre-rotation lifetime spend), `0.0` on first spawn; without it, rotating the log away would zero the running total. Readers that report spend (providers OQ3 untrusted-cost marking, watchtower budget rules, web-UI spend telemetry) consume `cost_usd` as the shared contract — see the §5 status row. **(v2.1, UL2)** Subagent (Task-tool) token spend rolls into the parent turn's `result` cost, so this same sum already captures it — no separate subagent accounting and no new schema field; `--max-budget-usd` and the C2 token ceiling bound it like any other turn cost (§6 worker-subagents).

**Additive-schema rule (v2.1, M1).** `fleet.json` grows by ADDING fields, never renaming/removing. **Readers default any missing field** (an old record lacking `cost_baseline`/`max_budget_usd`/`setting_sources` reads as `0.0`/`null`/`null`); **writers preserve unknown fields** (round-trip a record without dropping keys a newer/older writer added). This is what keeps the single writer safe as later work extends the schema (providers adds a profile field; watchtower adds mute/cursor state) without a migration step or a format-version gate.

**Spawn-immutable settings (v2.1, M5).** `max_budget_usd` and `setting_sources` are recorded at spawn and are IMMUTABLE thereafter, exactly like `mode`/`cwd`/`model`. They exist because both underlying CLI flags are per-`--print`-invocation: EVERY launch path (spawn, send-when-idle resume, respawn) MUST re-pass them from the registry, or the budget cap and the foreign-hook source restriction silently evaporate on turn 2 (§6, §11).

Not yet deleted — do not treat this paragraph as a live build order for new probe work. `probe_liveness` (`bin/fleet.py`) returns three verdicts, never a bare alive/dead boolean; the `ACCESS_DENIED` marker + `Get-CimInstance` fallback, the never-demote-alive-unknown rule, the retry-once-before-`dead` transition, and `_doctor_check_unreadable_starttime` all exist in code. The one part of this paragraph still `[UNBUILT — owned by C2 hardening kernel item 9]` is the §5 doctor **elevation-mismatch** check. The stored `turn_pid`/`turn_pid_ctime` classify into exactly one of three verdicts:
- **alive-and-matching** — PID exists AND image is claude AND creation time matches `turn_pid_ctime` (± 2 s). Turn is live.
- **definitely-gone** — PID absent, or present with non-matching image/ctime (PID reuse). Turn ended. (PID reuse would otherwise misclassify a dead worker as working and let `interrupt` kill an innocent worker's turn — so the ctime match is load-bearing.)
- **alive-unknown** — the process exists but the query itself failed, OR the stored `turn_pid`/`turn_pid_ctime` is null/corrupt. Real case: a worker spawned from a higher-integrity/elevated terminal, probed by a `fleet status` running unelevated — `Get-Process` returns the object but `.StartTime` throws `Win32Exception`, so a naive probe reads empty stdout and wrongly concludes "gone." The launch path itself stores `None` on a ctime-query failure, another source of alive-unknown.

The probe separates these via a try/catch marker in the PowerShell script (optional `Get-CimInstance` fallback). **alive-unknown is never demoted to `dead`, and is never respawn-/kill-eligible without `--force`.** A missing pid/ctime or any query failure counts as not-live, but **any `working`→`dead` transition first retries the probe once after a short delay**; only a stable `definitely-gone` may demote. Operating rule: run fleet at the SAME elevation as the workers it spawned — otherwise `taskkill` is access-denied and `interrupt` cannot clean up; `fleet doctor` checks for the elevation mismatch. Accepted residual: a genuinely-crashed turn whose PID is unreadable stays `working` (alive-unknown) until a retry succeeds or an operator runs `respawn --force` — fail-safe toward NOT launching a second live claude. Per-OS query-failure modes are carried into portability OQ2 (`docs/specs/portability.md`, owned separately).

**Boot identity gates the ctime compare (v2.2, F33) `[UNBUILT — owned by C4 port-adapter-a]` — PRESCRIPTIVE, no part of this paragraph ships today** **(now CANCELLED — native-substrate pivot 2026-07-13; see v2.2 appendix banner)** (`grep -rn "boot_id" bin/ tests/ --include=*.py` → no matches, at `9b5954d`). A `turn_pid_ctime` is only a process identity if it names an absolute instant. On Windows and macOS it does: Windows stores `Get-Process().StartTime.ToUniversalTime()`, an absolute FILETIME; macOS stores `ps -o lstart=`, rendered from `kp_proc.p_starttime`, a wall-clock `timeval` captured once at fork. On **Linux** it does not: the only per-process start time is `/proc/<pid>/stat` field 22, **clock ticks since boot** — the same number recurs at the same boot offset after every reboot. Two processes started at the same offset of two different boots therefore compare equal, the probe returns **alive-and-matching for a PID the fleet does not own**, and `_interrupt_worker` `killpg`s a live, unrelated process group (invariant 7). Fleet closes this by storing, next to `turn_pid_ctime`, the boot the ticks are relative to:

- **`turn_pid_boot_id`** — `string | null`. **Writer:** the launch path, which stamps `PLATFORM.boot_identity()` into the same record-commit that writes `turn_pid`/`turn_pid_ctime` (one atomic write per launch, under `fleet.lock`). The launch-path read is **guarded and hoisted above the `Popen`**, and an `OSError` there stamps `null` rather than escaping — see F33 *"The launch-path read"* (F33-R1); a launch may never fail, and may never orphan a live `claude`, because a boot id was unreadable. **Readers:** `probe_liveness` alone; every other consumer only forwards it out of the record. **Value:** on Linux, the contents of `/proc/sys/kernel/random/boot_id` — a kernel-global UUID regenerated at each boot, `0444 root:root`, unprivileged-readable, unaffected by `hidepid` (which gates only `/proc/<pid>/` directories). On Windows and macOS it is **always `null`, and that is normal, not legacy** — those OSes' ctime already carries intrinsic cross-boot identity, so there is nothing for a boot token to add.
- **A `null` on a record never decides anything by itself.** The discriminator is `PLATFORM.boot_identity()` read **fresh, on the running machine, at probe time** — a second, independent observation, not an overload of the stored value. `boot_identity()` returns `None` **if and only if** the platform defines no boot identity; a Linux read failure raises `OSError` and is never encoded as `None`. So: current `None` (Windows/macOS) ⇒ the gate is inert and today's logic runs unchanged; current non-`None` + stored `null` (a Linux record written before this field existed) ⇒ **alive-unknown**, because the boot the ticks belong to is unknowable; current non-`None` + stored non-`null` + unequal ⇒ **definitely-gone**, the one verdict in this spec justified by positive proof rather than by absence; equal ⇒ fall through to the ±2 s tick compare, unchanged.
- **Behavior on a registry written before the field existed:** exactly the **additive-schema rule** above — readers default the missing key to `null`, the single writer preserves it on round-trip, no migration step and no format-version gate (the `spawned_by` / `limit_reset_at` / `limit_kind` precedent). On Windows such a record is indistinguishable from a current one. On Linux it reads alive-unknown until the worker's next launch stamps the field; alive-unknown is never demoted to `dead` and is never respawn-/kill-eligible without `--force`, so the degradation authorizes neither a second live claude (invariant 7) nor a `killpg`.

Full derivation, the five failure cases, and the grep-enumerated core-edit list are appendix **F33**. The Linux `boot_id` file and the boot-relative nature of field 22 are verified facts, not assumptions: `docs/specs/portability.md` `## Boot identity (SPEC §4 F33)`.

**Status recompute is scoped (v2.1, B1).** Recompute derives status from the log tail + probe ONLY for records already in status `working` with a **non-None `turn_pid`**. Every other state is sticky or pre-claim and recompute leaves it untouched:
- **`dead` is sticky** — the sole exit is `fleet respawn`. An operator `kill` is terminal; recompute never resurrects it. (Without this, a `result` line in the log tail flips a killed worker back to idle and `fleet clean` becomes unreachable — SMOKE-D, pinned by `tests/test_core.py:497`.)
- **`attached` is sticky** — the sole exit is `fleet release` (or `respawn --force`). Recompute never demotes attached → idle; demoting it would let a `send` launch a resume turn against a session that may still hold a live TUI, breaking one-live-claude.
- **(v2.1, UL1) `limited` is sticky-until-reset** — a worker parked on a Claude plan usage-limit hit is EXEMPT from the `working`→`dead` crash path, exactly as `dead`/`attached` are sticky. Recompute never demotes a `limited` record to `dead` on a definitely-gone probe: a limited worker legitimately holds no live claude — the turn ended at the plan wall, not in a crash, so "no live PID + no `result`" is the EXPECTED shape, not a crash signal. The sole exits are `now >= limit_reset_at` → **resume-eligible** (a `fleet resume-limited` launch flips it to `working`) or `respawn --force`. See the Usage-limit park paragraph below.
- **`working` + `turn_pid = None` = launch-in-flight (pre-claim)** — written under `fleet.lock` before `Popen` by every turn-starting path. A **FRESH** pre-claim (real launchers stamp `turn_pid` within seconds) is NOT demoted by recompute — this window is exactly what enforces one-live-claude against a concurrent send/attach. A **STALE** pre-claim, whose `last_activity` is older than `LAUNCH_CLAIM_MAX_AGE_SECONDS` (600 s), IS demoted to `dead` by recompute (shipped `recompute_status`): the launcher died between the pre-claim write and stamping the PID (a killed CLI, a `Popen` that never returned), and the record would otherwise pin `working` forever. A stale-pre-claim → dead record is recoverable via ANY dead-path lever (`respawn`, `kill`, `attach --force`, `clean`), not only `respawn`.
- **Classification uses the last SUBSTANTIVE stream event**, scanning backwards past `type:"system"`, hook lines, and junk — NOT the literal last stdout line. §6 mandates `--include-hook-events`, so hook/system lines land AFTER the `result` line on every real launch (SMOKE-A); keying on the literal last line would mark every healthy completed turn as dead. For an in-scope `working` record: last substantive event is `result` → idle; no `result` and probe is definitely-gone → dead (crashed turn); probe alive-unknown → stays `working` (retry once first). **(v2.1, UL1)** a no-`result` turn is NOT unconditionally crash-dead: the classifier first tests the usage-limit signal (below) — a definitely-gone probe with a limit-shaped turn-end parks `limited`, not `dead`.

**Usage-limit park (v2.1, UL1 — new resilience feature).** A `claude -p` turn can end not because the worker finished or crashed but because the ACCOUNT hit a Claude plan usage limit — either the **5-hour rolling session window** or the **weekly cap** (operator is on Max 20x, which meters both). This is ORTHOGONAL to per-worker `--max-budget-usd` (UL-OQ5): a worker can hit the plan wall at `$0.00` of its own budget — a plan-wide wall *with a reset time*, NOT a dollar/token ceiling the operator set. It must classify DISTINCTLY from the budget-trip branch (the §11 runaway worker-level cap) and from crash-dead. The fleet PARKS such a worker in the new status `limited` (distinct from `working|idle|attached|dead`), preserving its unfinished intent — mailbox + journal untouched, exactly as `send`/`respawn` already rely on — and records the reset horizon in two additive fields:
- `limit_reset_at` — ISO-8601 UTC instant the window resets, or `null` when the signal carried no parseable reset time (then the worker is parked `limited` with an unknown horizon and surfaced for an operator-set reset — never auto-resumed blind).
- `limit_kind` — `session_5h | weekly | null` (`null` when the kind is undetermined).

Both obey the **additive-schema rule** above (readers default a missing field to `null`; the single writer preserves them on round-trip — no migration gate). A `limited` record is sticky-until-reset (recompute exemption above). When `now >= limit_reset_at` the worker is **resume-eligible**; the resume launch (§5 `fleet resume-limited`) drains its mailbox + journal into a fresh turn through the ordinary `fleet_lock`-guarded launch path (invariants 6 single-writer-registry, 7 one-live-claude — pre-claim under the lock, universal drain), flipping it `limited → working` and re-passing the spawn-recorded `max_budget_usd`/`setting_sources` like every launch.

**Detection contract (UL-OQ1) — environment-dependent, §0.2.9 evidence + kernel-probe deferral.** Detection lives in the fleet.py turn-end classifier — the same launch/classify path that reads the stream-json tail to decide idle/dead — **NOT in a hook** (invariant 2 keeps hooks read-only and exit-0; the classifier is where §4 already derives state). **Evidence gathered (this machine, `claude` 2.1.204):** `claude --help` exposes NO usage-limit or reset-horizon surface — no `--limit`, no `--reset-at`, no usage/quota flag and no documented reset output field; the only budget-adjacent flag is `--max-budget-usd`, a per-invocation DOLLAR ceiling (the budget-trip a limited turn is explicitly NOT). So a reset horizon, if available at all, must be parsed from the turn-end signal itself, not read from a CLI flag. A real plan-limit wall CANNOT be forced on demand to observe the live signal — hitting it would burn the operator's actual plan usage — so the exact live signal is recorded: **`DEFERRED-TO-KERNEL-PROBE:` the C2 `harden-*` kernel (hardening item 11, which owns a live harness) MUST confirm, against a real or CLI-simulated usage-limit turn, (1) the exact turn-end signal — stream-json error event shape and/or stderr text and/or exit code a limit turn emits; (2) whether a machine-readable `limit_reset_at` is present and in what format; (3) the `session_5h` vs `weekly` discriminator — and PIN the parser to that observed signal before the auto-resume path is built.** No signal is invented and presented as verified here. Until the kernel probe confirms, the spec builds against the **conservative fallback** (intent §3 clause 2): a turn that ended ERRORED (no `result` stream event) AND whose stderr is **limit-shaped** (matches a usage-limit / reset-time pattern) is classified `limited-suspected` → parked `limited`, with `limit_reset_at`/`limit_kind` set from the signal if parseable else `null`, and **SURFACED for operator confirmation** rather than trusted blindly. An errored no-`result` turn whose stderr is NOT limit-shaped stays the ordinary crash-dead path (§11) — the fallback never swallows a genuine crash as a park, and never parks on a guessed signal.

## 5.1 Destructive-command guard (provenance)

**Origin (2026-07-09, real data loss).** A `claude -p "/fleet:status"` probe, granted `Bash(fleet:*)` by a read-only slash command, judged four dead workers untidy, killed a fifth that was `working`, and swept all five journals. It never exceeded its permissions. Narrowing the grant fixed that command. This guard fixes the CLI, so the next over-helpful agent — especially one under `bypassPermissions`, where **no permission prompt is ever shown** — cannot destroy a worker it never owned without saying so out loud.

**Provenance.** `spawned_by` (additive, §4 schema; readers default `null`) records the `CLAUDE_CODE_SESSION_ID` of the session that ran `fleet spawn`, or `null` when a human shell did. It is **immutable across respawn**: respawning someone else's worker must not transfer ownership, or `respawn --force` then `kill` launders the guard in two steps.

**Rule.** `kill`, `clean` and `respawn` refuse to act on a **foreign** worker unless `--yes` is passed. A worker is foreign when `spawned_by` differs from the caller's session **or is unknown** — every record written before this field existed has no owner, and the guard fails toward asking. A worker you spawned needs no confirmation.

**Scope.** The guard applies to **Claude sessions only**, detected by `CLAUDE_CODE_SESSION_ID`. A human at a plain shell is never guarded: fleet has always been a human-driven CLI and interposing friction there would break every existing script for no safety gain. The threat model is an agent, not an operator.

**No interactive prompt, deliberately.** An agent's Bash tool has no stdin to answer one with (`input()` raises `EOFError`, and the operator sees a traceback rather than a refusal). `isatty()` cannot distinguish the cases on Windows anyway: Git Bash's `/dev/null` is `NUL`, a **character device**, so `fleet kill x < /dev/null` reports a tty. An agent passes `--yes`; there is nothing to prompt.

**Worker turns strip `CLAUDE_CODE_SESSION_ID`** from the child environment (§6). The child `claude` stamps its own; an inherited one would make a worker running `fleet kill` indistinguishable from the manager that spawned it, letting it retire its siblings silently.

**`interrupt` is exempt.** It ends a turn, not a worker, and the transcript survives. Only `kill` / `clean` / `respawn` destroy or rotate state.

**Recovery.** `state/events.jsonl` is append-only and untouched by `clean`. Every `spawned`/`respawned` event carries `session_id` + `cwd`, which is enough to `claude --resume` a swept worker. The claude session survives; the journal does not.

**The universal drain rule (closes all stranding races):** every turn launch — spawn, send-when-idle, respawn — first drains `mailbox\<sid>.md` into the prompt. A message can land in the mailbox after the Stop hook drained but before the process exits; it is picked up by the next launch, and `fleet status` flags `idle+mail` so the manager notices workers with undelivered mail. **(v2.1, M3)** the drain rule and the `idle+mail` flag are both keyed to the CURRENT sid, so mail stranded on a RETIRED sid (a send racing a respawn sid-swap) is invisible to them — that window is closed by the send-serialization rule (send row) and any residue caught by doctor's "orphaned mailbox" check, not by this drain rule.

## 6. Worker turn launch (Windows detail)

**[SUPERSEDED — native-substrate pivot 2026-07-13]** This section's detached-`Popen` launch plumbing and PID-based liveness verdicts (§4's three-way probe, `turn_pid`/`turn_pid_ctime`) are superseded by native `--bg` dispatch through the daemon (`claude agents`), which owns process hosting and liveness; fleet keeps only the semantic layer (mailbox, budget, journal, outcome capture) on top. See `docs/superpowers/specs/2026-07-13-native-agents-pivot-design.md` §3/§5 and `docs/specs/native-substrate.md`. MOVED, not deleted — kept below for history; do not build against it.

```
claude -p --output-format stream-json --verbose --include-hook-events
  [--resume <sid> | --session-id <sid>]
  --settings C:/proga/claude-fleet/state/worker-settings.json
  [--permission-mode acceptEdits|dontAsk|plan | --dangerously-skip-permissions]
  [--model <m>] [--max-budget-usd <x>]
```

- **Prompt via stdin, never argv** (32K CreateProcess limit; preamble+task+journal easily exceeds it). `claude -p` reads the prompt from stdin; Popen writes prompt then closes stdin. Never leave stdin open/inherited — non-TTY open stdin causes a 3 s stall + `Warning: no stdin data received` on stderr every turn.
- `--verbose` is mandatory with `-p --output-format stream-json` (CLI errors without it).
- Launched via `subprocess.Popen`, `creationflags=DETACHED_PROCESS|CREATE_NEW_PROCESS_GROUP`, `cwd=<worker cwd>` (resume invariant §2), stdout → `logs\<name>.jsonl`, **stderr → separate `logs\<name>.err`** (stderr in the jsonl breaks parsing). Parsers additionally skip non-JSON lines defensively. (Machine gotcha: never background via Git-Bash `&` — reaped. Win32 flags or Start-Process only.)
- First turn `--session-id <uuid>`; later turns `--resume <sid>`.
- Record `turn_pid` + creation time. **Concurrency guard:** never two claude processes on one session — send-while-working writes mailbox only; attach refuses while working.
- The `--settings` path is the machine-local `state\worker-settings.json` INSTANCE (`fleet init` renders it from the git-tracked template — §7/§14), NEVER the repo-root template; passing the template would ship placeholder paths.

**Launch-sequence contract (v2.1, U17 — item 11).** Every turn-starting path (spawn, send-when-idle, respawn) follows ONE ordered sequence; the spec pins it because a partial launch must never strand a claim or corrupt the registry (invariants 6, 7):
1. **Precondition:** fleet is initialized — `state\worker-settings.json` exists and is fresh; a launch against a missing/stale instance is a `doctor`-flagged precondition failure, not a silent foreign-hook-only turn.
2. **Pre-claim under `fleet.lock`:** write `status=working`, `turn_pid=None` (the launch-in-flight window §4), release the lock BEFORE `Popen` — this window blocks a concurrent send/attach from racing the launch (invariant 7).
3. **DOA window → `TurnLaunchError`:** `Popen` can fail (bad cwd, missing claude, OS refusal) or the child can die immediately (a DOA turn that never writes a `result`). On a failed/DOA spawn the launcher raises `TurnLaunchError` and does NOT leave the pre-claim behind: it RESTORES the prior status and clears the claim (claim-restore + registry rollback, under `fleet.lock`), so a failed launch is invisible to the next recompute rather than a pinned-`working` ghost.
4. **Threaded stdin write:** the prompt (preamble+task+journal+drain, easily >32K) is written to the child's stdin on a SEPARATE thread, then stdin is closed — a synchronous write can deadlock when a large prompt fills the pipe before the child starts reading; never leave stdin open/inherited (the 3 s no-stdin stall above).
5. **Commit the turn PID:** on a successful spawn, write `turn_pid`+`turn_pid_ctime` under `fleet.lock`, the ctime read back from the SAME OS process query used at check time (never the launcher wall clock), so the ±2 s liveness match is sound (§4).

**Permission modes** (default `dontask` when `--mode` is omitted — auto-deny outside the allowlist, never `bypass` by default) (per-task, validated against real CLI choices `acceptEdits, auto, bypassPermissions, manual, dontAsk, plan`):
- `bypass` → `--dangerously-skip-permissions` (trusted grind in known repo)
- `accept` → `--permission-mode acceptEdits`
- `dontask` → `--permission-mode dontAsk` (middle ground: auto-deny outside allowlist rather than stall)
- `plan` → `--permission-mode plan`
- `omit` → no flag; print mode *denies* un-allowlisted actions (does not hang) — worker likely ends early; only for workers you intend to attach to.

**Foreign hooks policy:** worker repos' own `.claude/settings.json` hooks and global plugin hooks (caveman, codex stop-gate) merge into worker turns. v1 policy: leave them on (project hooks are usually wanted, e.g. lint gates); if a foreign Stop-hook fights fleet's turn-end model, spawn supports `--setting-sources` passthrough to restrict sources. Documented in the skill; not automated in v1. **(v2.1, M5)** `--setting-sources` is per-`--print`-invocation like `--max-budget-usd`: spawn records the chosen value as `setting_sources` (§4, spawn-immutable) and EVERY launch path re-passes it. Without this the restriction evaporates on turn 2 while the foreign Stop-hook fires on every turn — so the passthrough is stored, not one-shot.

**Worker subagents (v2.1, UL2).** A worker MAY use its own SUBAGENTS — the native Claude Code Task/Agent tool, which spawns an ephemeral child agent to run a scoped task and return a result. This is SANCTIONED and DOCUMENTED, not newly mechanized: the capability ships with the CLI, and the fleet launch path does not disable it. Probe evidence (appendix F32): `claude --help` exposes `--agents <json>` (DEFINE custom agent types) and an `agents` subcommand that manages *background* agents (`--bg`) — but the built-in Task/Agent tool needs NO flag to exist, and `build_turn_argv` passes no tool-restriction while the rendered `worker-settings.json` restricts no tools, so a headless `claude -p` worker has subagents AVAILABLE BY DEFAULT. No fleet flag enables or gates them in v1. Consequences the fleet reconciles:
- **Invariant 7 preserved** — a subagent is a child agent spawned WITHIN the worker's turn, NOT a competing `--resume` on the worker's `session_id`; the worker's session stays singular (one live claude per session). Subagents run under their own ephemeral context and do not concurrently write the worker's transcript in a resume-breaking way.
- **Doctor never false-positives them** — the `claude agents` inventory (§5 doctor row) lists BACKGROUND-agent sessions (the `--bg` dispatched persistent sessions), which ephemeral Task-tool subagents are NOT: they are in-turn children invisible to that inventory, so they never register as fleet-unknown `claude agents` sessions and no doctor exemption code is owed (F32/UL2-OQ2).
- **Permission inheritance (SECURITY)** — subagents inherit the worker's permission posture: a `bypass` worker's subagents run `--dangerously-skip-permissions` too, widening a bypass worker's blast radius. v1 policy: DOCUMENT, do not mechanically restrict — a bypass worker is already trusted (cross-ref the permission-modes + foreign-hooks discipline above).
- **Cost already bounded** — subagent tokens roll into the parent turn's `result` cost, so the §4 cost model (`cost_usd` = baseline + Σ result costs), `--max-budget-usd`, and the C2 provider token ceiling already bound them; no new schema field.
- **Steering unchanged** — fleet's PostToolUse/Stop mailbox hooks fire on the WORKER (main-agent) tool boundaries; subagent-internal tool calls are nested and not individually mailbox-checked, so a mid-turn `send` still lands when the subagent returns and the main agent hits its next boundary (§7).
- **Foreign agents (like foreign hooks)** — a worker inherits the target repo's `.claude/agents/*` plus global plugin agents; available subagent TYPES are the union of built-ins + target-repo + global. v1: no manager curation (mirrors the foreign-hooks v1 policy above).

**PostCompact registration (v2.1, M6).** The template above will gain a THIRD entry, `PostCompact`, wiring a hook that appends a compaction marker to the worker's own journal (`state\journals\<name>.md`) so a context compaction is visible to a fresh session. The event name is real: the installed Claude Code CLI ships a `PostCompact` hook event (verified against claude 2.1.203 in `docs/reviews/SPEC-REVIEW-2026-07-08.md`; this machine now runs 2.1.204). Because the journal is keyed by NAME but a hook receives only `session_id`, the PostCompact script resolves `session_id`→`name` via a read-only, failure-tolerant registry read, falling back to a sid-keyed path or skipping on any failure (§4 hook write boundary). The kernel-3 spec text (the actual script, plus teaching `fleet doctor`'s hook-event lint the new name) lives in `phase1-hardening-kernels.md`, owned by `stub-inject-kernels-webui` — cross-referenced here, not edited.

Both hooks read stdin JSON (`session_id` is provided), check `mailbox/<session_id>.md`:

- **PostToolUse:** non-empty → claim by `os.replace` to `<file>.claimed.<pid>`, emit `{"hookSpecificOutput": {"hookEventName": "PostToolUse", "additionalContext": "<MANAGER MESSAGE>\n..."}}`, delete claimed file. Worker sees it between tool calls — seconds-level mid-turn latency. Missing/empty → exit 0 silently.
- **Stop:** mailbox non-empty → claim, `{"decision": "block", "reason": "<contents>"}` → turn continues with the queued instruction. Empty → allow stop. Honor `stop_hook_active` politely; **no custom loop counter** — Claude Code force-allows stop after 8 consecutive blocks natively, and continuation requires fresh manager mail anyway.
- **Hook scripts are exception-proof:** everything in try/except, always exit 0 on any error (uncaught tracebacks surface stderr into the worker's transcript as noise). The except path may append ONE best-effort diagnostic line to `state\hook-errors.log` (append-mode, no lock — §4 hook write boundary) before exiting 0; failure to write that line is itself swallowed. Parallel PostToolUse invocations race on the claim — `os.replace` loser catches FileNotFoundError and exits 0; concurrent append during rename can raise PermissionError on NTFS — claimer retries once then defers to the next boundary.
- Multiple sends append to one mailbox file (small `open(..., "a")` writes); first hook to claim drains all.
- Hooks are additive via `--settings` — managed projects' own settings still apply; fleet never edits project files.
- Silent-failure alarm: settings validation errors are swallowed in print mode, so `fleet doctor` validates `worker-settings.json` and smoke-tests both hook scripts; `fleet peek` shows whether hook events appear in the stream (`--include-hook-events`).

## 8. Worker rules preamble

`fleet spawn` prepends a standard preamble to every task:

- You are fleet worker `<name>` in `<cwd>`; manager messages arrive mid-task marked `<MANAGER MESSAGE>` — treat as user instructions.
- Maintain journal at `C:/proga/claude-fleet/state/journals/<name>.md` (create early; update each milestone): goal, done, in-progress, blockers, next steps. Must be enough for a fresh session to continue. (Journal lives in fleet state, not the project repo — managed repos stay git-clean.)
- End every turn with a compact result summary (changed, verified, blocked).
- No servers/watchers outliving the turn without recording PIDs in the journal.
- **(v2.1, UL2)** Subagents (the Task/Agent tool) are available; use them freely — but any child process a subagent starts is still bound by the rule above (no servers/watchers outliving the turn without recording PIDs in the journal).

The journal is the context backbone: `fleet respawn` = fresh context, zero lost state.

## 9. Hybrid interaction model (human ↔ worker)

1. Observe anytime: `fleet peek` (mid-turn, reads live log).
2. Steer anytime: `fleet send` — mailbox mid-turn, resume when idle.
3. Take over: `fleet attach <name>` → terminal tab (wt or PowerShell fallback), full TUI, whole history.
4. Hand back: close tab, `fleet release <name>`.

**Attach asymmetry (stated, not hidden):** the attached TUI runs WITHOUT `--settings`, so fleet hooks don't fire during attach — mailbox mail queues untouched until the next headless turn. `fleet status` shows attach age + pending mail; `doctor` nags attaches older than a few hours. One live claude process per session, ever: attach refuses while a turn runs (`--force` interrupts), sends never spawn turns while attached.

## 15. Terminal surface (Phase 1.6)

Full spec: `docs/specs/terminal-surface.md`. A statusline, `/fleet:*` slash commands, a SessionStart briefing, and a plugin package — four DERIVED views over one read-only derivation, `fleet.status_snapshot()`.

The three rules the rest of this spec implies but never had to state, now that a view refreshes on a hot path:

- **A view never takes `fleet.lock`, never probes a PID, and never writes.** `status_snapshot()` reads `fleet.json` + `mailbox/` and stops. `fleet status` (no flag) keeps its authoritative recompute; `fleet status --stale-ok` is the view path (invariants 6, 9).
- **A view reports registry corruption; it does not quarantine.** §11's quarantine (`fleet.json.corrupt.<ts>` + `registry_corrupt` event + exit 1) is a WRITE and belongs to the single writer. A statusline refreshing every 10 s would loop on it, destroying operator evidence. Views print "registry unreadable" and exit 0.
- **A view flags, it never launches.** A `limited` worker past its reset horizon renders `resume-eligible`; the resume itself stays the explicit `fleet resume-limited` sweep (invariant 1).

One launch-path consequence: `launch_turn` stamps `FLEET_WORKER=<name>` into the worker's child environment so the SessionStart hook — which a globally-enabled plugin fires in EVERY session on the machine — suppresses itself inside workers.

Nothing in §1–§14 changes. Every surface is optional: the CLI works fully with the statusline uninstalled, the plugin absent, and the hook unregistered.

**The defect.** `probe_liveness` decides "is the process bearing `turn_pid` still the turn we launched?" by comparing a stored ctime against a freshly-read one, ±2 s (`bin/fleet.py:637`). That is sound only if the ctime names an absolute instant. Windows' does (absolute FILETIME). macOS' does (`kp_proc.p_starttime`, a wall-clock `timeval` captured once at fork — `[UNVERIFIED — no macOS host; verify in port-posix-smoke]`). **Linux's does not.** The only per-process start time is `/proc/<pid>/stat` field 22, in clock ticks since boot:

```
[VERIFIED — WSL Ubuntu] awk '{print $22}' /proc/self/stat -> 16469 ;
  getconf CLK_TCK -> 100 ; cat /proc/uptime -> 164.70 2582.76
  => 16469/100 = 164.69s, i.e. exactly "now, since boot". No absolute instant is encoded.
```

So a stored Linux ctime is a boot **offset**, and every reboot recreates every offset. The collision condition degrades from "two processes started within 2 s of the same absolute instant" (a coincidence in the irrecoverable past — measure-zero, never recurs; the residual already accepted at `bin/fleet.py:597-603`) to "**two processes started within 2 s of the same boot offset**", which **every reboot recreates**, and which is *densest exactly at boot*: the entire userspace boot spans 1.4–2.3 s at `CLK_TCK=100` — inside one ±2 s window — and ROADMAP Phase 2's logon-triggered manager spawns workers inside that burst. The failure direction is **false `alive`**, the worst one: `recompute_status` reports a dead worker `working`, and `_interrupt_worker` — which gates purely on `pid_alive` (`bin/fleet.py:3115`) — proceeds to `kill_process_tree(pid)` (`:3118`) on a live process group the fleet does not own. **Invariant 7.** (A synthetic-epoch ctime built from raw ticks, the fix-wave-1 answer, is NTP-immune by construction and still carries no boot identity — it inverts F1's false-`gone` into this false-`alive`. `/proc/stat`'s `btime` is `wall_now − uptime`, moves under a clock step, and was correctly rejected.)

**The decision.** A fifth platform-adapter method supplies a per-boot token; core stores it and compares it. `boot_identity(self) -> str | None`, on **both** adapter classes, inside the `# === PLATFORM ADAPTER START/END` block (`bin/fleet.py:191-342`):

| Adapter | Returns | Why |
|---|---|---|
| `_WindowsPlatform` | `None`, always | `Get-Process().StartTime.ToUniversalTime()` is an **absolute** FILETIME — it already carries intrinsic cross-boot identity, so a boot token adds nothing. `[VERIFIED — this machine, py -3.13 read-only: fleet.PLATFORM.get_process_info(os.getpid()) → ('python', datetime(2026,7,9,21,19,3, tzinfo=utc))]` |
| `_PosixPlatform`, `sys.platform == "darwin"` | `None`, always | `ps -o lstart=` renders `kp_proc.p_starttime`, captured once at fork from the wall clock — absolute, hence self-identifying across boots. `[UNVERIFIED — no macOS host; verify in port-posix-smoke]` |
| `_PosixPlatform`, otherwise (Linux) | contents of `/proc/sys/kernel/random/boot_id` | A kernel-global UUID regenerated at each boot. `[VERIFIED — WSL Ubuntu: ls -l → -r--r--r-- 1 root root; cat → 53b5856e-50a5-4cdf-b283-944c2f619d7d; wc -c → 37; two reads in separate processes minutes apart → identical]` |

Reading it costs no privilege and no `/proc/<pid>` visibility: it is `0444`, world-readable, and lives under `/proc/sys/`, which `hidepid` does not gate (`hidepid` restricts other users' `/proc/<pid>/` directories only). `[VERIFIED — WSL Ubuntu: cat /proc/mounts | grep ' /proc ' → proc /proc proc rw,nosuid,nodev,noexec,noatime; the path is not under /proc/<pid>/]`. It is **kernel-global, not namespaced**: `[VERIFIED — WSL Ubuntu: unshare -r --fork --pid --mount-proc cat /proc/sys/kernel/random/boot_id → the same UUID, exit 0]`.

**The wire shapes, and why none of them is overloaded.** The two prior fix waves both died of a value that meant different things to different callers — the `(name, None)` sentinel meant "alive, ctime unknown" to `probe_liveness` and "here is a ctime" to `launch_turn` (FW2-R1); `stored_boot_id=None` meant both "the caller omitted it" and "this record's field is null" (FW2-R5). This design introduces no third overload, by three separate constructions:

1. **`boot_identity() -> str | None`.** `None` means **exactly one thing**: this platform defines no boot identity. It is a static property of the OS, never a runtime outcome. A Linux read failure raises `OSError`; it is never encoded as `None`. `probe_liveness` catches that `OSError` and returns `"unknown"` — the fail-safe verdict, not a silent fall-through to the very tick compare this finding exists to gate.
2. **`turn_pid_boot_id: string | null` in the record.** A stored `null` never decides anything on its own. What it *means* is decided by `PLATFORM.boot_identity()` read **fresh, on the running machine, at probe time**. This is the whole design: the ambiguity is resolved by a second, independent observation rather than by overloading the stored value. Hence `null-because-this-OS-has-no-boot-id` (Windows/macOS, where a `null` is **normal**) and `null-because-the-record-predates-the-field` (legacy Linux) are distinguished without any sentinel beyond `None` — and no rule "stored null → alive-unknown" is ever stated unconditionally, which is precisely the rule that bricked fix wave 2.
3. **`stored_boot_id` is a required keyword-only parameter** on `probe_liveness` / `pid_alive` / `recompute_status` — **no default**. Omitting it at any call site is a `TypeError` at that call site, not a value. This is FW2-R5's disease cured by removal: the parameter that needed an `_UNSET` sentinel no longer exists on the adapter, and on core it cannot be omitted silently. `get_process_info` grows **no production parameter at all**, which also moots FW2-R3 (the Windows adapter is untouched) and leaves every injected one-argument `get_process_info=lambda pid: ...` test double working.

**What the stamped `null` then means, walked through the case table below rather than asserted.** `boot_identity()` raising at all implies Linux: `_WindowsPlatform` and the macOS branch return `None` as a static property of the OS and have no failure mode. **Accepted residuals.** (1) **Containers.** `boot_id` is kernel-global, so a containerised fleet reads the *host's* boot_id and a container restart without a host reboot does not move it; a plain PID-namespace restart still leaves `/proc/<pid>/stat` field 22 host-boot-relative, so tick values cannot collide with a stale record's — but `CLONE_NEWTIME` would defeat that. (2) **The intra-boot PID-reuse residual is unchanged and is not narrowed here.** Once case (ii)/(iii) return before the tick compare, the surviving collision needs one boot to allocate a pid, free it, wrap the whole pid space, and land the reused pid's `starttime` within ±2 s of the original's. The **structural** argument is what carries: within one boot, field-22 ticks increase monotonically and are never revisited, so a pid can only recur after a full wrap. The **quantitative** argument was previously stated as "a wraparound consumes far more than 2 s of ticks", which is not a fact about ticks — it is a fact about `pid_max`, a sysctl (F33-R3). Stated as the bound it actually is: reuse inside the window requires a **sustained fork rate ≥ `pid_max` / 2 s**, because every one of `pid_max` allocations must occur between the original's start and the reused pid's, and that interval is ≤ 2 s. At the measured value that is **2,097,152 forks/s** `[VERIFIED — WSL Ubuntu, cat /proc/sys/kernel/pid_max → 4194304; uname -r → 6.18.33.2-microsoft-standard-WSL2]`. At the conservative `32768` floor (a plausible unraised kernel default `[UNVERIFIED — verify in port-posix-smoke]`) it is **16,384 forks/s**, held for two seconds — a 128× smaller margin than the WSL number suggests, and the reason this spec quotes the formula rather than one box's constant. The reused pid must additionally pass the `_ALIVE_IMAGE_NAMES` filter (`bin/fleet.py:597-603`). Genuinely the same class as that site's accepted residual — *"a reused pid landing on an unrelated cmd/node/claude process within 2s of the recorded ctime"* — and by the same structure, not by analogy. The ±2 s tolerance stays. **`port-posix-smoke` records the box's `pid_max` alongside its `boot_id`**, so the bound is evaluated against the machine fleet actually runs on. ## Invariants touched (F33)

The F33 boot-identity gate touches two of the nine numbered architectural invariants:
- **7 one-live-claude-per-session** — the finding exists to keep a cross-reboot tick collision from reading `alive` on a PID the fleet does not own. Both null cases resolve to `alive-unknown`, which is never demoted to `dead` (so no `respawn` double-launch) and for which `pid_alive` is `False` (so no `kill_process_tree`); the mismatch case resolves to `gone` **before** the tick compare runs.
- **8 platform-adapter-only OS branching** — `boot_identity()` is a fifth adapter method; core stores and compares an opaque string and never learns what a boot id is. The `sys.platform == "darwin"` discriminator sits inside `_PosixPlatform`, within the marker-delimited block the lint excises. Verified against the lint's actual scan, not assumed.

## Invariants touched (UL1)

The UL1 usage-limit park+resume feature (F31 above) touches three of the nine numbered architectural invariants:
- **6 single-writer registry** — `fleet resume-limited` and the `limited`-park write go through `fleet.py` under `fleet.lock`; the new `limit_reset_at`/`limit_kind` fields are single-writer like every other registry field.
- **7 one-live-claude-per-session** — a parked (`limited`) worker holds NO live claude process; the resume launch obeys the pre-claim window (`working`+`turn_pid=None` under the lock before `Popen`), so resume never starts a second live claude on the session.
- **2 exit-0 hooks** — usage-limit DETECTION lives in the fleet.py launch/classify paths, NEVER in a hook (hooks stay read-only and exit-0 per M6); no hook writes the `limited` state or the reset fields.

## Invariants touched (UL2)

The UL2 worker-subagents capability (F32 above) touches four of the nine numbered architectural invariants — all PRESERVED, none modified:
- **7 one-live-claude-per-session** — a subagent is an ephemeral child agent spawned WITHIN the worker's turn, never a second `--resume` on the worker's `session_id`; the worker's session stays singular and its transcript is not concurrently written in a resume-breaking way.
- **6 single-writer registry** — subagents never write `fleet.json`; only `fleet.py` does, and a worker's subagent use is invisible to the registry.
- **2 exit-0 hooks** — unchanged: mailbox hooks still fire only on the worker's main-agent boundaries and stay exit-0; subagent-internal tool calls are nested and not individually hooked.
- **1 daemonless launch** — subagents are in-turn children that end with the turn; they add no resident process and require no auto-wake.
